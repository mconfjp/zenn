# 書籍執筆テンプレート

技術書典などのイベント向けに書籍を執筆するときのフォーマット集。
ORM歴史本プロジェクトの実際の作業から抽出したもの。

## フェーズ

書籍執筆は以下のフェーズで進める。各フェーズでやること、使える雛形・スキルをまとめる。

### 1. 企画構成

**やること:** AIと壁打ちしながら、書籍全体の企画と章立てを決める。
各章のコンセプト・ストーリーライン・パンチラインを明確にし、既存素材と調査が必要なものを洗い出す。

- 雛形: [[YYYY.M.D_企画_説明]]
- 命名: `YYYY.M.D_企画_[説明].md`
- 例: `2026.2.20_企画_歴史本の構成.md`

### 2. 企画構成レビュー

**やること:** 人間が企画をレビューする。
AIが書いた企画の鼻につく表現を直したり、構成を組み替えたり、「わかってること / わからんこと / 悩み」を整理する。章ごとにやるのがよい。

- 雛形: [[YYYY.M.D_企画構成レビュー]]
- 命名: `YYYY.M.D_企画構成レビュー_[章名].md`
- 例: `2026.2.21_企画構成レビュー_RDBの歴史.md`

### 3. 雑に書く

**やること:** 人間が一旦雑に書き出してみる。
きれいに書く必要はない。書いてみることで「わかっていること」と「わからないこと」が明確になる。ここで出た「わからんこと」がフェーズ4のネタ補強の入力になる。

- 雛形: [[YYYY.M.D_雑書き_分野_トピック]]
- 命名: `YYYY.M.D_雑書き_[分野]_[トピック].md`
- 例: `2026.2.22_雑書き_歴史_RDB_先史古代.md`

### 4. ネタの補強

**やること:** 「3. 雑に書く」で人間が雑に書いた文章をもとに、テーマ分割とネタの補強をする。
箇条書きをコンポーネント（意味のまとまり）に分解し、各コンポーネントにテーマを与え、ネタが足りないところを調査して補強する。

- スキル: `/material-check [ファイルパス]`
	- 箇条書きをコンポーネントに分解
	- 各コンポーネントのネタ充足度を評価
	- 不足分をWebSearchやvault内資料から調査
	- 「3. 雑に書く」で出力されたファイルの「AIによるネタ補強後」セクションに結果を追記

### 5. 箇条書き

**やること:** 人間が補強されたネタを取り込み、箇条書き原稿にまとめる。
フェーズ4で追記された「AIによるネタ補強後」セクションから必要なネタを選び、本文側に組み込んでいく。

- 雛形: [[YYYY.M.D_箇条書き_章名_セクション]]
- 命名: `YYYY.M.D_箇条書き_[章名]_[セクション].md`
- 例: `2026.2.22_箇条書き_RDBの歴史_先史時代.md`

### 校正（フェーズ5の後）

- スキル: `/ai-check [ファイルパス]`
  - AIっぽい表現（`**`記号、単調なリズム、説明書口調、事なかれ主義、抽象語・万能語、変な比喩）を指摘

## その他のファイル

雛形は用意しないが、執筆中に必要になるファイル。

### 気持ちファイル

作業中の感情や気づきを記録する。構造は最小限。

```
# これはなに
[一行説明]

# 気持ち
- [箇条書きで感想・気づき]
```

- 命名: `YYYY.M.D_[プロジェクト名]のきもち.md`

### 章別作業インデックス

章ごとの作業進捗を管理する。各章に「企画構成レビュー / 調査ネタだし / 箇条書 / メモ」のサブセクションを設ける。プロジェクトごとに構造が異なるので雛形はなし。

- 命名: `YYYY.M.D_[イベント名]で[書名]を出すルート.md`


## ファイル命名規則

- 日付形式: `YYYY.M.D`（ゼロ埋めなし。例: `2026.2.21`）
- Obsidianの `[[ファイル名]]` リンクで相互参照する
