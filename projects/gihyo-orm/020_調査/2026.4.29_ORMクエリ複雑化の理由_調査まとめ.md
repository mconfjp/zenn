# ORMが変なクエリを吐く理由 - 調査まとめ

## これはなに

2026.4.29 に実施した調査のまとめ。
「ORMから発行されるクエリが、思ったより複雑な形（サブクエリを使ったり）になっている理由を解き明かしたい」という問いに対して、広く事例を収集した。
この後、全て読んで壁打ちに進む予定。

---

## パターン一覧

### パターン1｜コレクション取得でJOINが使えない問題

- 1対多でJOINすると行が重複する
  - User1人にAddressが3つあれば、SQLの結果は3行になる
  - ORMはオブジェクトを返さないといけないので、この重複を除去する必要がある
- 重複を避けるためにサブクエリ戦略が生まれた
- これがあらゆる「変なクエリ」の根本原因のひとつ
- 参考：[[『One-To-Many relationship gets duplicate objects without using any trick』要約]]、[[『How to Avoid Duplicate Records in Hibernate／Spring Data JPA』要約]]

---

### パターン2｜カルテシアン積爆発（Django / Doctrine）

- 複数のコレクションを同時にJOINすると、行が掛け算で増える
- Doctrine の例：
  - 商品1件 × カテゴリ3 × 画像4 × タグ5 × レビュー10 = **600行**（商品1件なのに）
- Django の例：
  - 店舗 × 部門 × 従業員 = 2500 × 10 × 500 = **1,250万行**
- ORMが内部でJOINを使ってリレーションを取得するとき、意図せずこれが起きる
- 参考：[[『The Cartesian Product Trap in Django ORM』要約]]、[[『Doctrine ORM：How I Escaped the Cartesian Product Trap』要約]]

---

### パターン3｜ページネーション + JOINで全件取得する（Hibernate）

- コレクション（1対多）をJOIN FETCHしながらページネーションすると、HibernateはSQLにLIMITをつけられない
  - 「1つのUserに3つのAddressがあれば3行返る」＝行数とオブジェクト数がズレるため、正しいページ数が計算できない
- 結果：**LIMITなしで全件取得してメモリでページング**
  - 大量データで致命的なパフォーマンス問題になる
  - 警告ログ `HHH90003004` が出る
- 参考：[[『Efficient Collection Fetching and Pagination with Hibernate 6 and Spring Data JPA』要約]]

---

### パターン4｜N+1問題の解決策としての3戦略（SQLAlchemy）

- N+1を避けようとした結果、ORMは複数の戦略を持っている
  - **Joined Eager Loading** — JOINで一気に取得。1対多では重複問題が発生
  - **Subquery Eager Loading** — オリジナルクエリをサブクエリに埋め込んで関連データを別取得。複雑になるがコレクションに有効
  - **Select IN Loading** — `WHERE id IN (...)` で関連データを取得。現代ORMでは主流
- どの戦略を選ぶかで発行されるSQLが全然変わる
- 参考：[[『Relationship Loading Techniques — SQLAlchemy 2.1 Documentation』要約]]

---

### パターン5｜includesが勝手に戦略を切り替える（ActiveRecord）

- Railsの `includes` は「状況を見て」2つの戦略を自動的に切り替える
  - 条件なし → `preload`（別クエリ2本）
  - WHERE/ORDER条件に関連テーブルが絡む → `eager_load`（LEFT OUTER JOIN 1本）
- **開発者が意図していないのに、コードを少し変えただけでSQLが全然変わる**
- 参考：[[『Understanding Rails Eager-Loading』要約]]、[[『Preload, Eagerload, Includes and Joins』要約]]

---

### パターン6｜PrismaはそもそもJOINしない設計だった

- Prismaは長らく「DBでJOINせず複数クエリ発行してアプリ側でマージ」という設計をとっていた
  - 理由1：DBエンジン（PostgreSQL/MySQL/SQLite）間で挙動を統一できる（移植性）
  - 理由2：DBよりアプリサーバーのほうがスケールしやすいという設計思想
- 最近（v5/v6頃）になってようやく**JOINモード**が選べるようになった
  - PostgreSQLでは `LATERAL JOIN + JSON aggregation`
  - MySQLでは相関サブクエリ
- 参考：[[『Prisma ORM Now Lets You Choose the Best Join Strategy （Preview）』要約]]、[[『Database vs Application： Demystifying JOIN Strategies』要約]]

---

### パターン7｜PrismaのLATERAL JOIN + JSON aggregationの正体

- Prismaの新JOINモードがやってること（PostgreSQL）：

```sql
SELECT u.*, json_agg(o.*) as orders
FROM users u
LEFT JOIN LATERAL (
  SELECT * FROM orders WHERE user_id = u.id  -- u.id を参照している
) o ON true
GROUP BY u.id
```

- `LATERAL JOIN` は「外側の行ごとにサブクエリを実行する」もの。foreachループをSQL化したイメージ
- 結果をJSONで集約してアプリに渡す
- MySQLは `LATERAL JOIN` が制限されるため、代わりに相関サブクエリを使用
- 「複雑に見えるクエリ」が実はよく考えられた戦略という面白い対比
- 参考：[[『Prisma：Further JOIN on JOINed records』要約]]

---

### パターン8｜JPQLがサブクエリを書ける場所が限定的（Hibernate / JPA）

- HibernateのJPQL（オブジェクトクエリ言語）はサブクエリをWHERE / HAVING句にしか書けない
  - FROM句やSELECT句には書けない（SQLなら書ける）
- そのためSQL的には「FROM句にサブクエリ書けば一発なのに、JPQLの制約でWorkaroundなクエリが生成される」ことがある
- 参考：[[『Is your query too complex for JPA and Hibernate？』要約]]

---

### パターン9｜プロキシオブジェクトとセッション問題（Hibernate）

- Hibernateは遅延ロードのために「プロキシオブジェクト」を返す
  - 本物のエンティティではなく、アクセスされた瞬間にSQLを発行するダミーオブジェクト
- 問題：**DBセッションが閉じた後にプロキシのプロパティを触ると `LazyInitializationException` が飛ぶ**
  - 「え、オブジェクト取れてるのになんで？」という現象
- 解決策のひとつ「Open Session in View」（リクエスト中ずっとセッションを開けておく）も、不必要なクエリが大量発生するという別の問題を生む
- 「透過的に見える抽象化」が裏で何をやってるか見えないことによる典型的な事故
- 参考：[[『LazyInitializationException - What it is and the best way to fix it』要約]]

---

### パターン10｜継承マッピングでクエリ戦略が分岐（Hibernate / SQLAlchemy）

- OOPの「継承」をRDBに落とす方法が3種類あり、それぞれ生成されるクエリが全然違う

| 戦略 | テーブル構造 | 生成されるSQL |
|---|---|---|
| Single Table Inheritance | 1テーブルに全カラム | 常に `WHERE type = 'xxx'` が入る |
| Joined Table Inheritance | 親・子テーブル分離 | 常に `INNER JOIN` が入る |
| Table per Class | サブクラスごとにテーブル | ポリモーフィッククエリは `UNION` になる |

- `findAll()` 1行書いただけで、裏では `UNION` が走ることがある
- 参考：[[『Inheritance Mapping Strategies in Hibernate： Single Table, Joined, and Table per Class Explained』要約]]

---

### パターン11｜ソフトデリート + グローバルフィルターでサブクエリ爆発（EF Core）

- ソフトデリート（`deleted_at IS NULL` で論理削除）を実装すると、ORMはすべてのクエリに自動でフィルターを付加する
- JOINするたびに各テーブルにサブクエリが生成される：

```sql
SELECT * FROM Users u
  INNER JOIN (SELECT * FROM Orders WHERE deleted_at IS NULL) o ON ...
  INNER JOIN (SELECT * FROM Products WHERE deleted_at IS NULL) p ON ...
WHERE u.deleted_at IS NULL
```

- JOINが増えるほどサブクエリが増える。しかも全カラムのサブクエリ
- **「便利な機能を足すほどクエリが複雑になる」の典型**
- 参考：[[『EF Core In depth – Soft deleting data with Global Query Filters – The Reformed Programmer』要約]]

---

### パターン12｜UPDATE時に全カラムを上書きする（Hibernate）

- `user.name = "新しい名前"` と1フィールドだけ変えたつもりでも：

```sql
UPDATE users SET name='新しい名前', email='前と同じ', age=30, ... WHERE id=1
```

- 全カラムがUPDATEされる
  - 理由：Hibernateはエンティティごとに1つのUPDATE文をキャッシュしているから
  - 毎回「どのカラムが変わったか」を判定してSQL生成するより、全部更新する1文をキャッシュしておいた方が速い
- `@DynamicUpdate` をつけると変更カラムのみのSQLが出るが、今度は毎回SQLを動的生成するコストが発生する
- **パフォーマンス最適化のための設計が、別の問題を生む**
- 参考：[[『Hibernate Tips：Exclude unchanged columns from update』要約]]

---

### パターン13｜Identity Mapで古いデータが返る

- ORMはセッション内でロード済みのエンティティをメモリにキャッシュする（Identity Map）
  - 同じIDで2回クエリを打っても、2回目はDBを見ない
- 問題：**別のプロセス/別のリクエストがDBを更新しても、キャッシュが古いまま**
- Unit of Workパターンとセットで動いているため、「フラッシュ（flush）」するまで変更はDBに反映されない
  - 「保存したはずなのに反映されない」という現象はこれが原因のことが多い
- 参考：[[『Identity Map and Request Context』要約]]

---

### パターン14｜多対多の中間テーブルで意図しない挙動

- TypeORMのバグ実例：

```sql
-- 期待してたSQL
INSERT INTO user_role(user_id, role_id) VALUES (1, 2)
-- 実際に出たSQL
INSERT INTO user_role(user_id, role_id) VALUES (DEFAULT, DEFAULT)
-- → NULL, NULLが入る
```

- SQLAlchemyの罠：`session.delete(obj)` と `query.delete()` で挙動が違う
  - 前者は中間テーブルのレコードも消える
  - 後者は中間テーブルが残る
  - 同じ「削除」操作なのに
- 参考：[[『Many-to-many relationships broken when user-defined junction entity exists (0.3.26 regression) · Issue ＃11637 · typeorm／typeorm』要約]]、[[『Deleting by Query has a different behavior compared to deleting an ORM instance when using many-to-many mapping · sqlalchemy／sqlalchemy』要約]]

---

### パターン15｜COUNT / GROUP BYが過剰なSQLになる（MikroORM）

- `findAndCount()` でこんなSQLが出ることがある：

```sql
SELECT count(DISTINCT e0.id) FROM posts e0
  LEFT JOIN users ON ...
```

- 本来 `count(*)` で十分なのに `count(DISTINCT id)` になる
  - インデックスが効かずフルスキャンになる
  - 実測でほぼ1桁のパフォーマンス差になった報告がある
- 参考：[[『Generated count query is inefficient · mikro-orm／mikro-orm』要約]]

## 全体を貫く3つの軸

今回収集した事例を整理すると、3つの軸に集約できる。

| 軸 | 代表パターン |
|---|---|
| **インピーダンスミスマッチを解消しようとする** | カルテシアン積、継承マッピング、ポリモーフィック、コレクション×JOINの重複問題 |
| **利便性・安全性のために裏で何かしている** | Dirty Checking全カラムUPDATE、グローバルフィルター、楽観的ロック、Identity Map、includesの戦略切り替え |
| **DBエンジンの差を吸収しようとする** | PrismaのLATERAL vs 相関サブクエリ、JPQLの制約 |
