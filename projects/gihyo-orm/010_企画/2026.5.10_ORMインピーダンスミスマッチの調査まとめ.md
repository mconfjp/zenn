# ORMとインピーダンスミスマッチ - 調査まとめ

## これはなに

2026.5.10 に実施した調査のまとめ。
「インピーダンスミスマッチとは何か、何がどうズレているのか」を体系的に整理したい、という問いに対して、Hibernate公式・Wikipedia・Ted Neward論文・jOOQブログなどの一次情報を収集した。
記事1本目の「インピーダンスミスマッチ」セクションを書くための下地として使う。

---

## パターン一覧

### パターン1｜粒度のミスマッチ（Granularity Mismatch）

- オブジェクトモデルはDBより「細かく」作られる
  - Javaでは `Address`、`PhoneNumber`、`Money` を個別のクラスに分けることが自然
  - DBではそれらを `users` テーブルのカラムとして平たく格納する
- テーブル数よりクラス数のほうが多い、というのが典型的な状態
- **ORMの対応**：`@Embeddable`（JPA/Hibernate）、Embedded Value Object パターン
  - 複数の細粒度クラスを1テーブルの複数カラムにマッピングする
  - 例：`user.address.city` → `users.city`（別テーブルなし）
- 参考：[[『What is object-relational mapping』要約]]

---

### パターン2｜構造のミスマッチ（Structural Mismatch）

- OOは数学的には**有向グラフ**：オブジェクトが他のオブジェクトを参照する
  - 入れ子、特殊化（継承）、カプセル化が前提
- RDBは**フラットな行列**：テーブルと列、正規化が前提
  - 入れ子はなく、全ての関係はJOINで後から組み立てる
- オブジェクト界は「グラフをたどる」世界、RDB界は「表を結合する」世界
- 「グラフをそのままテーブルに落とせない」ことがミスマッチの源泉
- 参考：[[『Wiki：Object–relational impedance mismatch』要約]]

---

### パターン3｜同一性のミスマッチ（Identity Mismatch）

- **RDB**：同一性は主キーだけで定まる。同じPKを持つ行は「同じもの」
- **OOP**：同一性が2種類ある
  - オブジェクト同一性：`a == b`（同じメモリアドレスを指す）
  - 値の等値性：`a.equals(b)`（内容が同じ）
- 問題：同じDB行を複数のJavaオブジェクトで表現することが起きる
  - `user1 = findById(1)` と `user2 = findById(1)` は `user1 != user2`（別インスタンス）
  - でも `user1.equals(user2)` は実装次第で true になる
- **ORMの対応**：Identity Mapパターン
  - セッション内でロード済みのエンティティをキャッシュし、同じPKなら同じインスタンスを返す
  - → これがセッションをまたいだときに「古いデータが返る」問題を生む（別の調査まとめで詳述）
- 参考：[[『What is object-relational mapping』要約]]
	- [[『Wiki：Object–relational impedance mismatch』要約]]

---

### パターン4｜関連のミスマッチ（Association Mismatch）

- **OOP**：関連は**単方向の参照**として表現される
  - `Order` が `Customer` を持つ（`order.customer`）
  - 逆方向（`customer.orders`）は別途コードが必要
- **RDB**：外部キーは本質的に**双方向**（どちらの方向にもJOINできる）
  - `orders.customer_id = customers.id` で `orders` → `customers` も `customers` → `orders` も取れる
- 問題：
  - OOPでは多重度（1対1、1対多）がコードから読み取りにくい
  - 双方向関係をOOで持つには両側にコードが必要（整合性を保つのが難しい）
  - 「参照を持つ」と「外部キーで参照される」という方向が逆になることもある
- 参考：[[『What is object-relational mapping』要約]]
	- [[『Object-Relational Impedance Mismatch』要約]]

### パターン5｜型システムのミスマッチ（Type System Mismatch）

- SQLの型とプログラミング言語の型は1対1に対応していない
- **文字列**
  - SQL：最大長（VARCHAR(255)）・照合順序（collation）を持つ。末尾スペースを比較時に無視することがある
  - Java：`String` は長さ制限なし。照合はソートルーティングで
- **NULL**
  - SQL：三値論理（true / false / NULL）。`NULL = NULL` は `NULL`（trueではない）
  - Java：`null` はオブジェクト参照の不在。`null == null` は `true`
  - → NullPointerException vs NULL伝搬という根本的に違う問題が発生
- **数値**
  - SQL：`DECIMAL(10,2)` のように精度・スケールが固定
  - Java：`int`、`float`、`BigDecimal` など精度管理の考え方が異なる
- **日付・時刻**
  - タイムゾーン、精度（マイクロ秒 vs ミリ秒）の扱いがDBとJVMで異なる
- 参考：
	- [[『Wiki：Object–relational impedance mismatch』要約]]
	- [[『Types - Doctrine Database Abstraction Layer (DBAL)』要約]]

### パターン6｜継承・多態のミスマッチ（Inheritance Mismatch）

- **OOP**：クラス継承が自然なパラダイム（Animal → Dog, Cat）
- **RDB**：継承を第一級でサポートしない。テーブルは「型」を持てない
- 継承階層をテーブルに落とす方法が3種類あり、選択によってSQLが変わる

| 戦略 | テーブル構造 | 生成されるSQL |
|---|---|---|
| Single Table Inheritance (STI) | 1テーブルに全カラム | `WHERE type = 'Dog'` が常に付く |
| Joined Table Inheritance (JTI) | 親・子テーブル分離 | 常に `INNER JOIN` が必要 |
| Table per Class | サブクラスごとにテーブル | ポリモーフィッククエリは `UNION` |

- `findAll()` 1行書いただけで裏では `UNION` が走ることがある（別の調査まとめで詳述）
- 参考：[[『What is object-relational mapping』要約]]

### パターン7｜データナビゲーションのミスマッチ（Navigation Mismatch）

- **OOP**：オブジェクトグラフをたどる
  ```java
  order.getCustomer().getAddress().getCity()
  ```
- **RDB**：JOINを使って一度に関連データを取得する
  ```sql
  SELECT city FROM orders
    JOIN customers ON ...
    JOIN addresses ON ...
  ```
- OOPのグラフ走査をそのままDBアクセスに使うと、N+1問題が発生する
  - `orders` を100件取得 → `order.getCustomer()` を100回呼ぶ → 100クエリ発生
  - 合計101クエリ（1本の取得 + 100本のN+1）
- RDBでは「クエリ数を最小化してlatencyを制御する」が設計原則
  - OOPの「たどる」発想と根本的に相容れない
- **ORMの対応**：Eager Loading（N+1を避けるためにJOINで先読み） vs Lazy Loading（たどるときに初めてDB取得）
  - → このトレードオフがEager/Lazy戦略、PrismaのJoin戦略選択などに繋がる（別の調査まとめで詳述）
- 参考：
	- [[『What is object-relational mapping』要約]]
	- [[『The Object-Relational Impedance Mismatch: Why Your Java Objects and Database Tables Don't Play Nice』要約]]

### パターン8｜操作パラダイムのミスマッチ（Operation Mismatch）

- **SQL（RDB）**：集合指向（set-based）・宣言的
  - 「何を取得するか」を宣言する
  - データ操作演算子はすべてのデータに共通（SELECT / WHERE / JOIN / GROUP BY）
  - 操作の結果もリレーション（表）
- **OOP**：オブジェクト指向・命令的（imperative）
  - 「どうやって処理するか」を記述する
  - 操作はクラスごとに固有のメソッドとして実装される
  - 操作の結果はオブジェクト
- ORMはこの橋渡しをしているが、「オブジェクト操作」を「SQL操作」に変換する過程で無駄なクエリが生じる
  - 例：`user.name = "新名前"` という1フィールドの変更が `UPDATE users SET name=..., email=..., age=...` という全カラムUPDATEになる（Hibernate Dirty Checking問題）（別の調査まとめで詳述）
- 参考：[[『Wiki：Object–relational impedance mismatch』要約]]

### パターン9｜インピーダンスミスマッチはミスリードである（歴史的背景）

- 「インピーダンスミスマッチ」という言葉は、ある意味でミスリードを含んでいる
- **RDBの登場（1970年、E.F. Codd）が何をもたらしたか**
  - それ以前の階層型・ネットワーク型DBでは、データへのアクセスパスがスキーマに物理的に焼き付いていた
  - Coddのリレーショナルモデルは「**データの論理構造と物理構造を分離した**」
  - → 物理的な格納方法を変えずに、論理的なデータ構造を変えられるようになった
- **これがOOPの発展を可能にした**
  - アプリケーションコードが「データをどこにどう置くか」を気にしなくてよくなった
  - OOPは「オブジェクトで思考し、オブジェクトでデータを持つ」ことができるようになった
- **逆説**：RDBとOOPは互いに独立して発展したからこそ、根本的なモデルの違いが生じた
  - どちらかが間違っているのではなく、「独立して最適化された2つのシステムが、協調しなければならなくなった」
- **jOOQの視点**：本当のミスマッチはモデル間にあるのではなく、「SQLの設計目的（クエリと新規関係の生成）とORMの使い方（部分的なオブジェクト具現化・変更保存）のズレ」にある
  - リレーショナルモデルと階層型モデルは「矢印の方向が逆なだけで本質的に同じものを表現している」とも言える
- **Ted Newardの絶望的な視点**：ORMは「コンピュータ科学のベトナム」
  - 始めは簡単に見える、しかし進むうちに解決不可能な複雑さに陥る、明確な出口戦略がない
  - 最終的な結論は「OかRのどちらかを選ぶ」（オブジェクトをやめるか、RDBをやめるか）
- 参考：
	- [[『There is no Such Thing as Object-Relational Impedance Mismatch』要約]]
	- [[『Object-Relational Mapping is the Vietnam of Computer Science』要約]]

## 全体を貫く3つの軸

今回収集した事例を整理すると、3つの軸に集約できる。

| 軸 | 代表パターン |
|---|---|
| **モデルの構造的差異**（グラフ vs 表） | 粒度・構造・継承のミスマッチ |
| **同一性・関連の表現の差異** | 同一性・関連・型のミスマッチ |
| **操作パラダイムの差異**（宣言的 vs 命令的、集合 vs グラフ走査） | ナビゲーション・操作のミスマッチ |

「インピーダンスミスマッチはミスリード」という視点（パターン9）は、これら全ての軸を横断する歴史的背景であり、記事の序文で語れるテーマ。
