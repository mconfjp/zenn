# 整理
- 先行と遅延というふたつで捉えるのがなんか違ってる
- というよりも、直積の爆発するJOINクエリに対してどんなケースでどんな対応ができるか？　というくくりで考えた方が良い
- どういうときに機能改修まで考慮するか？　をしめさないとだめ
	- 相関サブクエリだとどうせN1相当に重たいからなおせみたいな
	- ネットワークの分は軽いよ
- 技術駆動な分類が意味をなさない理由
	- Eager, Lazy、Eagerはさらにいくつか、という分類だと、なんか違う
		- 多分、オブジェクトを引き回すとか、集計とか、そういうアプリケーションの観点がかけている
		- あと俺が今回気がついたEagerとLazyが対応しているとか、割とこれは入り口で、そのあとに幾つかの細かいものが入ってきていて、そこまで行くとこのメンタルモデルは邪魔な気がする
		- 「メモリに一気に載せる or 必要な時に取ってくる」この構図はまあ間違ってない
			- それぞれN1か直積OOM問題かを選んでる
			- どっちでもないケースが結構多いので、単純な二項対立と思ってると良くないってことではあるか。。
			- じゃあ一旦この対立軸は残しておいて、より細かく見るとこう、というのを出すか
- 分類
	- ジョイン先で絞りたいなら先行ロード
	- ジョインするまえに絞れるならセレクトINで二度引きするか、サブクエリか
	- 条件分岐入って使うかわからないならlazy
	- row単位の集計があるなら、相関サブクエリ
	- それでもデータ爆発するならページング、機能改修
# 整理
- まずJOINクエリには問題がある
	- ロードするデータがデカくなりすぎる
- ふたつ
	- でかくなりすぎてOOM
	- でかいのを細かく処理していくとN+1
- ここに対してORMが提供してるのがEagerとLazy
	- Eagerは先行ロードでN+1をふせぎ、Lazyは遅延ロードでOOMを防ぐ
- ただしこれは巨大データのリスクをメモリで受け止めるかパフォーマンスで受け止めるかのトレードオフでしかなく、大きすぎるという本質的な課題を解決はしない
- そこへの解決策は
	- 機能改修
	- ページネーション
	- Select in
- ってかんじ

ToOneとかも含めていいけど、ただそれを入れてもこの文章のテーマからは外れるのでノイズだなって思った。
ここで面白いのは、JOINクエリの持つ課題の見える化と、それがEagerとLazyとに対応するってのと、しかしこれはリスク受容の手法でしかなくて本質的な解決は別にあるということだと思う。この３段階の最後に対応方法の細かいユースケースみたいなものが出 てきてて、単純なEagerとLazyはそもそもリスク受容できるレベルのリスクに対しての方策でしかない。受容できないものはまずはSelectIN、そして機能改修やページネーションで対応するってなりそう。
