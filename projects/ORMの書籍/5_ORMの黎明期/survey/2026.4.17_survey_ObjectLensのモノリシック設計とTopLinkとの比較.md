# survey: ObjectLensのモノリシック設計とTopLinkとの比較

## 調査目的

ParcPlace ObjectLensがなぜ「モノリシック」と評されるか、TopLinkのData Mapper的分離との設計上の違いを理解する。

---

## メンタルモデル

### 1. ObjectLensの設計：アプリケーション中央集権型

ObjectLensは**アプリケーションクラスに1つの `dataModelSpec` メソッド**を置き、そこにアプリケーション内の全エンティティのDB→オブジェクトマッピングを一括定義する方式。

```smalltalk
" アプリケーションクラス上の1つのメソッドが全マッピングを持つ（VisualWorks 2.5の実際の記法）"
^#(#LensDataModel
    #setDatabaseContext: #(#Oracle7Context #username: '' #environment: '' )
    #structureTypes: #(
        #(#LensStructureType
            #memberClass: #SomeEntity
            #setVariables: #(
                #(#LensStructureVariable
                    #name: 'someField'
                    #column: #(#Oracle7TableColumn #name: 'SOME_FIELD' #dataType: 'varchar2' )
                    #privateIsMapped: true ) )
            #table: #(#Oracle7Table #name: 'SOME_TABLE' ) )
        ... " 全エンティティ分をここに列挙 " )
    #lensPolicyName: #Mixed
    #lensTransactionPolicyName: #PessimisticRR
    #validity: #installed )
```

※ `#{Lens.LensDataModel}` のような `#{}` 記法はVisualWorks 5i以降の名前空間記法。VisualWorks 2.5では `#LensDataModel` のようなシンボル形式。（出典：[[2026.4.18_COMPORA.ST_VisualWorks2.5チュートリアルソース]]）

「モノリシック」の意味は：
- **集中定義**：1メソッドが全エンティティのマッピングを記述
- **継承の手動展開**（Prasse Slide 11）：ドメインクラスが継承関係にある場合（`Person → Employee` 等）、サブクラスの `LensStructureType` には親クラスの変数も含めて書き直しが必要。`Person` に変数を追加したら全サブクラス分を手動更新しなければならない。※ただしこれはPrasseの実案件（継承が多い年金ドメイン）固有の話であり、フラットなドメインモデルでは問題にならない
- **単一ソース原則がない**（Prasse Slide 14、主訴）：プロダクトファミリー（共通コアを持つ複数アプリ）を開発する場合、各アプリが `dataModelSpec` の完全なコピーを持ち続けることになる。共通部分を1か所で管理する仕組みがないため、共通ドメインの変更が全コピーへの手動伝播を要求する。※これもPrasse/Collogia社という1チームの状況。単一アプリの範囲ではGUIツールが全生成・全管理するため問題にならない

### 2. TopLinkの設計：クラス単位の分離型（Data Mapper）

TopLinkは**クラスごとにDescriptor（記述子）オブジェクト**を持ち、それをProjectが集約する方式。

- `Project` → `Descriptor`（クラスごと） → `Mapping`（フィールドごと）
- ドメインオブジェクト自体はDB知識を持たない
- Descriptorは独立したメタデータオブジェクトであり、アプリケーションクラスに埋め込まれない

これがFowlerが後にData Mapperパターンと命名したアーキテクチャの原型。

### 3. 対比の本質

| 観点 | ObjectLens | TopLink |
|---|---|---|
| マッピング定義の場所 | アプリケーションクラスの1メソッド（全エンティティ分） | クラスごとのDescriptorオブジェクト |
| 継承の扱い | 手動展開（サブクラスに親変数も書く） | Descriptorレベルで継承サポート |
| ドメインクラスとの結合 | アプリが一括管理（ドメインクラスは関与しない） | ドメインクラスから分離された別レイヤー |
| 拡張性 | プロダクトファミリーで重複爆発 | クラス単位で追加 |

## ObjectLensのアーキテクチャ詳細（一次資料より）

### 4層構造

```
Layer 1: GUI層
  ├── LensMainApplication  (ApplicationModelのサブクラス、メインアプリ用)   [*1]
  └── LensDataManager      (ApplicationModelのサブクラス、データフォームウィンドウ用)   [*1]
          ↓ dataModelDefinitionSpec を定義
Layer 2: Data Model層
  └── LensDataModel   [*2]
        ├── LensStructureType（クラスごとのマッピング定義）   [*2]
        │     └── LensStructureVariable（フィールドごとのカラム対応）   [*2]
        └── LensGlobalDescriptor（グローバル設定）   [*1][*2]
          ↓ fromLiteralArrayEncoding: でデコード
Layer 3: ObjectLens runtime層
  ├── LensSession   ← 永続化操作のAPI窓口   [*1][*2]
  ├── LensContainer ← 永続オブジェクトコレクション管理   [*1][*2]
  └── LensProxy     ← 遅延ロード用プロキシ   [*2]
          ↓
Layer 4: EXDI層
  └── ExternalDatabaseSession（生SQL発行）   [*1]
```

- [*1] ParcPlace-Digitalk, VisualWorks 2.5 User's Guide (DS10005004), Ch.18（LensMainApplication/LensDataManagerの役割定義）, Ch.26（削除依存の記述）, Ch.25（LensSession/LensContainer/LensGlobalDescriptorのクラスリスト）
- [*2] Prasse 2006 SlideShare（https://www.slideshare.net/esug/modular-lenseng3）

削除依存：App Framework → ObjectLens core → RDBMS Interface の順で強制 [*1]

### `dataModelSpec` → `LensSession` の連携メカニズム

```smalltalk
" アプリクラスに自動生成されるメソッド（= dataModelSpec の実体）"   [*2]
primDataModelDefinition
    ^#( ... )  "LiteralArrayエンコードされた全エンティティのマッピング定義"

" LensDataModel がデコードして構造を保持 "   [*2]
LensDataModel fromLiteralArrayEncoding: aLiteralArray

" LensSession のCRUD API "   [*2]
aLensSession add: anObject.     " → INSERT/UPDATE "
aLensSession remove: anObject.  " → DELETE "
aLensSession select: [ ... ].   " → SELECT (Smalltalkブロックで条件指定) "
```

**連携フロー（[*2] より）：**  
`primDataModelDefinition` がLiteralArrayを返す → `LensDataModel` がデコードして構造を保持 → `LensGlobalDescriptor` にグローバル登録 → `LensSession` がマッピング情報を参照してSQL生成・結果をオブジェクトに変換

### Prasseが提案したモジュール化の核心（[*2] より）

| | Before（モノリシック） | After（Prasse案） |
|---|---|---|
| 定義メソッド | `primDataModelDefinition`（全クラス一括） | `dataModelStructureTypeSpecs`（クラスごとに収集） |
| 組み立て | LiteralArrayを直書き | `dataModelSpecForStructureTypeSpecs:` が動的組み立て |
| 基底テンプレート | なし | `dataModelTemplate` で共通部分を定義 |
| カスタマイズ | コピペ | `adaptDataModel: ldm` でオーバーライド |

## 使い方のコード例

出典はすべて [*2] Prasse 2006 SlideShare（https://www.slideshare.net/esug/modular-lenseng3）

### 素のObjectLens（モノリシック）

#### アプリクラスへのマッピング一括定義

アプリクラス（`LensMainApplication` のサブクラス）の1メソッドに全エンティティのマッピングを書く。

```smalltalk
MyApp >> primDataModelDefinition
    ^#(#{Lens.LensDataModel}
        #setDatabaseContext: #( ... )
        #structureTypes: #(
            #(#{Lens.LensStructureType}
                #memberClass: Customer
                #setVariables: #(
                    #(#{Lens.LensStructureVariable}
                        #name: 'name'
                        #column: 'CUSTOMER_NAME'
                        #privateIsMapped: true )
                    #(#{Lens.LensStructureVariable}
                        #name: 'createdAt'
                        #column: 'CREATED_AT'
                        #privateIsMapped: true ) )
                #table: 'CUSTOMERS' )
            #(#{Lens.LensStructureType}
                #memberClass: Order
                #setVariables: #( ... )
                #table: 'ORDERS' )
            " ... 全エンティティ分をここに列挙 " )
        #lensPolicyName: #Mixed
        #lensTransactionPolicyName: #PessimisticRR
        #validity: #installed )
```

#### LensSession を使った永続化操作

```smalltalk
" 登録（INSERT/UPDATE）"
aLensSession add: aCustomer.

" 削除（DELETE）"
aLensSession remove: aCustomer.

" 検索（SELECT）"
aLensSession select: [ :each | each name = 'Taro' ].
```

### Prasseのモジュール化版

#### クラスごとの自己定義（各ドメインクラスに定義）

```smalltalk
Customer >> primDataModelDefinition
    | type |
    type := LensStructureType new.
    type memberClass: self.
    type table: (Oracle7Table new name: 'CUSTOMERS'; owner: 'MYSCHEMA').
    type idGeneratorType: #userDefinedId.
    ^type
```

通常は自動生成。カスタマイズが必要な場合は `primLocalDataModelDefinitionChanges: type` をオーバーライドする。

#### アプリクラスでの組み立て（LensMainApplicationのサブクラス）

```smalltalk
" 全クラスの定義を収集する "
MyApp >> dataModelStructureTypeSpecs
    ^self dataModelStructureTypeSpecsFor: self dataModelClasses

MyApp >> dataModelStructureTypeSpecsFor: classColl
    ^(classColl collect: [ :cl | cl dataModelDefinitionSpec ]) asArray

" 共通テンプレート（DB接続情報・ポリシー等）"
MyApp >> dataModelTemplate
    ^#(#{Lens.LensDataModel}
        #setDatabaseContext: #(#{Oracle7Context} ... )
        #structureTypes: #()
        #lensPolicyName: #Mixed
        #lensTransactionPolicyName: #PessimisticRR
        #validity: #installed )

" テンプレートに収集したクラス定義を差し込んで完成させる "
MyApp >> dataModelSpecForStructureTypeSpecs: aColl
    | res |
    res := self dataModelTemplate copy.
    res at: 5 put: aColl.
    ^res

" アプリ固有の後処理（任意）"
MyApp >> adaptDataModel: ldm
    " ... アセンブル後のLensDataModelをカスタマイズ "
```

#### モノリシック版との対比ポイント

| 変更箇所 | モノリシック | Prasse案 |
|---|---|---|
| マッピング定義の場所 | `MyApp >> primDataModelDefinition`（全クラス一括） | 各クラス自身の `primDataModelDefinition` |
| 親クラスの変数変更 | 全サブクラスの記述を手動更新 | 該当クラスのみ更新すればよい |
| 新クラス追加 | `primDataModelDefinition` を編集してエントリ追加 | `dataModelClasses` に追加するだけ |

## ObjectLensとRAD文化（GUIツールによるコード自動生成）

### 調査の経緯

「dataModelSpecはGUIツールの出力先として設計されていたのでは」という仮説を立て、一次資料で検証した。

**資料の探索ルート：**
1. User's Guide (vw25ug.pdf) のCh.12でデータベースツール群の記述を確認 → Data Modeler・Canvas Composer・Mapping Tool・Query Editorの存在は確認できたが、Data ModelerがGUIかどうかは明示されていなかった
2. 「VisualWorks Database Tools Tutorial and Cookbook」（別ドキュメント）を探したが、ESUGアーカイブ・Internet Archive・Cincom等いずれにも見つからなかった
3. Internet Archiveの VisualWorks 2.5ソフトウェアパッケージ (VisualWorks2.5.zip) を展開 → `TUTORIAL/DATABASE/COMPORA.ST`（公式チュートリアルのSmalltalkソース、71KB）を発見
4. このソースファイルから決定的な証拠を得た [*3]

### 一次資料から確認できた事実（[*3] COMPORA.ST より）

**クラスコメントに自動生成であることが明記されている：**

```
CompanyApplication:
  "This class was automatically generated by the VisualWorks tools when creating
   the data model and main database application canvas."

Empinfo（ドメインクラス）:
  "This class corresponds to the database table named 'EmpInfo'.
   It is automatically generated when the database table is imported into the data model."

DeptTwoUI:
  "The windowSpec in this class was created almost exclusively by the VisualWorks UI tools
   and required no extra Smalltalk programming."
```

**`dataModelSpec` のコメントにGUIツール呼び出しが書いてある：**

```smalltalk
dataModelSpec
    "LensEditor new openOnClass: self andSelector: #dataModelSpec"
    <resource: #dataModel>
    ^#(#LensDataModel ...)
```

→ `LensEditor` というGUIツールがこのメソッドを「開いて編集する」ための呼び出しがコメントに埋め込まれている。

### 確認できた設計の全体像

VisualWorksには `XxxTool new openOnClass: self andSelector: #メソッド名` という慣用句がある。「このメソッドを編集するときはこのGUIツールを開け」という意味で、自動生成されたメソッドのコメントに埋め込まれる。COMPORA.STにはこのパターンが全ツール分揃っている [*3]：

| ツール | 担当するメソッド | GUIツールであることの根拠 |
|---|---|---|
| `LensEditor` | `dataModelSpec`（O/Rマッピング定義） | 同パターンの他ツールと同列 |
| `QueryEditor` | `ownQuery` 等（クエリ定義） | VisualWorksの一般知識で確実 |
| `UIPainter` | `windowSpec`（UIレイアウト） | VisualWorksの一般知識で確実 |
| `UIDefiner` | アクセサ・イベントハンドラ等 | 自動生成ツール |
| `UIMenuEditor` | `menuBar`（メニュー定義） | 同パターン |

`LensEditor` は `UIPainter`・`QueryEditor` と**同一の慣用句パターン**で登場する。これらがGUIツールであることはVisualWorksの一般知識として確実なので、`LensEditor` も同様にGUIツールと判断できる。

ObjectLensのワークフローは以下のように**完全にGUIツール主導**で設計されていた：

| 成果物 | 生成するツール | 確認状況 |
|---|---|---|
| ドメインクラス（Empinfo等） | Data Modeler（DBからインポート） | [*3] コメントに明記 |
| `dataModelSpec` | LensEditor | [*3] メソッドコメントに明記・同パターン論証 |
| UIクラス（DeptTwoUI等） | Canvas Composer / UIDefiner | [*3] コメントに明記 |
| クエリメソッド（ownQuery等） | QueryEditor | [*3] コメントに明記 |
| メニュー定義 | UIMenuEditor | [*3] メソッドコメントに明記 |

User's Guide の「**no required form building or Smalltalk programming**」[*1] という文言は、このアーキテクチャの設計目標を正直に示していた。

### dataModelSpecのモノリシック設計の再解釈

この事実を踏まえると、「全エンティティを1メソッドに一括定義」という設計は**欠陥ではなく、GUIツールが管理する単一ファイルとしての自然な帰結**だった可能性が高い。

- LiteralArrayというフォーマット → GUIツールが読み書きするためのシリアライゼーション形式
- アプリクラスに集中 → LensEditorが1箇所を開けば全マッピングを管理できる
- 手書きを想定していない → ツールが生成・更新するものであり、人が読むことを主目的としない

Prasseの「コピペ地獄」問題は、**ツールが想定していなかったプロダクトファミリーという使い方**で発生した。単一アプリの範囲では、ツールが全生成・全管理するので問題にならない。

[*3] ParcPlace-Digitalk, "VisualWorks 2.5" チュートリアルソースコード `TUTORIAL/DATABASE/COMPORA.ST`, 1995年9月28日
  - Internet Archive: https://archive.org/details/visualworks-2.5 → VisualWorks2.5.zip 内

## 論争点

### 「モノリシック」は欠陥か設計上の意図か

- 肯定側：GUIツール（LensEditor等）が自動生成・管理する前提での設計。ツールで使う分には「全マッピングを1箇所で管理」は利点。宣言的でパーサブルなLiteralArray形式はツール親和性が高い
- 否定側：2006年のPrasse講演が実際の問題を示す。プロダクトファミリー化すると `dataModelSpec` のコピペ地獄が発生し、単一ソース原則を破る
- 判定：**ツールで使う限り設計上の意図は成立していた。プロダクトファミリーという「ツールが想定しない使い方」で限界が露呈した**

### ObjectLensはTopLinkに「先行」したか

- maetlは「ORMのアイデアの起源はおそらくObjectLens（probably where these ideas originated）」と言及
- ただし現時点の最古一次資料はInfoWorld 1994年5月記事（VisualWorks 2.0発表時）
- TopLink for Smalltalkも1994年。どちらが先かは一次資料で未確定
- 「先行」の根拠はmaetlの発言のみで、商標や論文からは確認できていない
