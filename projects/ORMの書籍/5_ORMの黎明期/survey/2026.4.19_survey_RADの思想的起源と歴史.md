# survey: RAD（Rapid Application Development）の思想的起源と歴史

## 調査目的

ObjectLensやFileMaker・Microsoft Accessが同時期（1990年代前半）に出てきた背景として、RADという思想的潮流がどのように生まれ、何を目指していたのかを理解する。アジャイルとの接続も視野に入れる。

---

## メンタルモデル

### 1. RADはウォーターフォールへの反発として生まれた

1970年代に確立したウォーターフォール開発は、「要件は最初から明確、変化しない」という前提に立っていた。しかし現実には：
- 要件は開発中に変化する
- ユーザーが本当に欲しいものは作ってみないとわからない
- 大規模プロジェクトは計画通りに進まない

1980年代、この批判が高まる。Barry Boehm は1986年の論文でSpiralモデルを提唱し、**リスク分析とプロトタイピングの繰り返し**を開発の中心に置いた。これがRAD思想の知的土台になる。

→ 出典：Boehm, "A Spiral Model of Software Development and Enhancement", 1986（[MSU PDF](https://cse.msu.edu/~cse435/Homework/HW3/boehm.pdf)）

### 2. James Martinが1991年にRADを体系化した

IBMでの実務経験をもとに、James Martinが1991年に著書 *Rapid Application Development* を発表。RADの主要な要素：

- **反復的開発・プロトタイピング**：最初から完成品を作らず、動くものを見せながら要件を固める
- **JRP/JAD**（Joint Requirements Planning / Joint Application Design）：ユーザーを開発に積極的参加させるワークショップ手法
- **CASEツールの活用**：コード生成・画面設計などをGUIツールで自動化
- **タイムボックス**：固定した期間内に収まるよう機能をスコープする

→ 出典：Martin, James. *Rapid Application Development*. Macmillan, 1991.（[Internet Archive](https://archive.org/details/rapidapplication00mart)）

### 3. 1980年代のGUIと4GLがRADを技術的に可能にした

RADが登場できた背景には2つの技術的変化がある：

**GUIの普及（1980年代前半〜）**
- Macintosh（1984年）・Windows（1985年〜）がGUIを大衆化
- 「画面をコードではなく視覚的に設計できる」ツールが生まれる土台に

**4GL（第4世代言語）の台頭**
- SQLのような「何をしたいか」を宣言的に書く高水準言語
- 画面設計・レポート生成・DB操作を統合した開発環境（CASE tools）
- 「プログラミングなしでアプリを作れる」という夢の技術的根拠

→ 出典：[Rapid application development - Wikipedia](https://en.wikipedia.org/wiki/Rapid_application_development)

### 4. CASEツール運動との重なり（1982〜1993年）

1982年にCASE（Computer-Aided Software Engineering）という用語が生まれ、1980年代末には100社以上が200種類近いCASEツールを提供していた。「CADが機械設計を変えたように、CASEがソフトウェア開発を自動化する」という期待があった。

しかし1993年のGAO（米政府監査院）レポートは冷や水を浴びせた：

> "Little evidence yet exists that CASE tools can improve software quality or productivity"

「プログラミング不要」という夢は、限定的な領域（DBアプリ・UI自動生成等）でしか実現しなかった。

→ 出典：[Rapid Application Development history - Buzzclan](https://buzzclan.com/digital-transformation/rapid-application-development/)

### 5. 1990年代に「速さ」が最重要課題になった背景

Kneuper（2017）はp.49で1990年代の変化をこう説明している：

> "Some of the main developments in software engineering during the 1990s were the growing importance of **time-to-market** of software, partly due to the **emergence of the World Wide Web**, and the resulting need to better adapt to unclear and changing requirements."

WWWの登場によってソフトウェアのリリース速度が競争優位に直結するようになり、「重厚な計画→長期開発→一括リリース」というウォーターフォールのサイクルが市場に追いつけなくなった。これが1990年代前半に軽量プロセス・RADツールが一斉に登場した直接の動機。

この「速さ」の追求はプロセス手法（RAD・Scrum等）だけでなく、**ソフトウェア構築の道具そのもの**にも波及した。DBアプリケーション構築がその典型で、GUIツールがコードを自動生成することで「Smalltalkプログラミング不要でDBアプリを作れる」環境が1990年代前半に実現している（後述）。

→ 出典：Kneuper 2017, p.49（[IEEE Xplore](https://ieeexplore.ieee.org/document/8047358/)）

### 6. RADからアジャイルへの系譜

RADの「反復・プロトタイピング・ユーザー参加」という発想は、その後の開発方法論に直接引き継がれた：

| 年 | 出来事 |
|---|---|
| 1991 | James Martin, *Rapid Application Development* |
| 1994 | DSDM（Dynamic Systems Development Method）登場——RADの乱立を統一する方法論として |
| 1995 | Scrum |
| 1996 | Crystal Clear・Extreme Programming（XP） |
| 1997 | Feature-Driven Development（FDD） |
| 2001 | アジャイルマニフェスト——DSDM・Scrum・XP・FDDらの代表者が集結 |

アジャイルマニフェストは新しい方法論ではなく、1990年代に乱立していた軽量プロセスの**共通原則を言語化したもの**。RADはその出発点の一つ。

→ 出典：[Agile Manifesto History](https://agilemanifesto.org/history.html)・[DSDM Wikipedia](https://en.wikipedia.org/wiki/Dynamic_systems_development_method)

## ObjectLens・FM・Accessとの接続

| 製品 | RAD的特徴 |
|---|---|
| FileMaker Pro（1990） | カードUIのGUI操作でDBアプリを作れる。プログラミング不要 |
| Microsoft Access（1992） | Windowsのウィザード・GUIでテーブル・フォーム・レポートを生成 |
| Visual Basic 1.0（1991） | GUIをドラッグ&ドロップで設計するRADツールの代表格 |
| ObjectLens（1994） | LensEditor等GUIツールがdataModelSpec・UIクラスを自動生成。「no extra Smalltalk programming」が設計目標 |

これらはすべて「1991年前後にRAD思想が最高潮に達した時期」に登場している。Martin本・VB・DSDM・Access・ObjectLensは同じ時代の空気を吸っている。

特にDBアプリケーション構築は、RADの「プログラミング不要」思想が最も具体的な形で実現したドメインの一つだった。ObjectLensのチュートリアルソース（COMPORA.ST, 1995）には「required no extra Smalltalk programming」と明記されており、GUIツールによる完全自動生成が設計目標だったことが確認できる。→ 出典：[[2026.4.18_COMPORA.ST_VisualWorks2.5チュートリアルソース]]

## 論争点

### 「RADは本当に速かったのか」

- 肯定側：単純なDBアプリ・フォームアプリの範囲では劇的に速くなった（FMやAccessの普及がその証拠）
- 否定側：CASEツールは期待ほど機能しなかった（1993年GAOレポート）。複雑なビジネスロジックには対応できず、「ツールで作れる範囲」の狭さが露呈した

### 「RADとアジャイルはどう違うのか」

- 連続説：アジャイルはRADの延長。「反復・ユーザー参加・動くものを優先」という核心は同じ
- 断絶説：RADはツール自動化による生産性向上が主眼（CASEツール依存）。アジャイルはプロセスと人間の協働が主眼。ツール依存から人間中心へのパラダイムシフトがある
