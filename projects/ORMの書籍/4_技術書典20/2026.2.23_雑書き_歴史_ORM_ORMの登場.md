## 4. ORMの歴史（4,000字）

>**コンセプト**: RDB（データを概念にした）とOOP（プログラムの抽象度を上げた）のインピーダンスミスマッチを解消するためにORMが登場した

- あー多分ここに至るまでに、OOPとRDBで思想をちゃんと全面に出していかないとダメなんだろうなあ
	- そもそも二つの出自が全然違うんだよなあ
	- その違いをもっとこう、明確にここで示してあげたい
		- まあ難しくいうとインピーダンスミスマッチなんだろうけど、、しかしあんまり理解できてない、しっくりきてない
- それはそうとして、技術的な進歩についてはだいぶ調べ尽くしてるはず
	- ORM前史
	- ふたつのパラダイム
	- ORMの登場
	- ORMの隆盛
- 課題
	- さっぱりしすぎるかも
	- OOPとRDBの思想的な違いを説明しておきたいのと、あとは技術的なものに限られちゃうから、ここも実務の世界の話を入れておきたい
### 2. ORMの登場（2,000文字）
	- 1994年 TopLink for Smalltalk：最初のORM、商用
	- 1996年 TopLink for Java
	- 2002年 Oracle買収 / Hibernate登場：商用→OSSへの転換
	- 2006年 ActiveRecord / Rails：フレームワーク標準装備へ
- わかること
	- 各種製品がいつ出たのか
	- 周辺の情報
		- MySQL4.0、Java
- わからんこと
	- 利用状況かなーーー
	- いつ頃完全に馴染んだ？
		- 1994に登場して、フレームワーク標準になったのはいつが初め？
	- TopLinkが最初どこまで機能を載せていたのか

### この章の意味
- ORMの歴史を書く
	- これは、永続化レイヤー抽象化の歴史の、かなり袋小路というか、ずっとこう発展しきったところ
	- RDBとOOPという前提の上で、どう抽象化がなされたのか
	- その前提なしだった場合、どうなるんだ
- データとロジックの確定
	- データは変わらず、ロジックは変化する
		- ロジックの変化を資本主義が求めてくる
		- 振る舞いとデータの整合性
	- インピーダンスミスマッチが起きる
	- 解決するためのORM
- DDDとかの現代に繋がるところ
	- Javaが出てきて、変化の中でORMが求められて、
	- これが結局なんなんだろうか、、、
		- RDBは保存をこうやって解決した、OOPはロジックがこうなった、ORMはその間にある負債を解消した？　してはないが
