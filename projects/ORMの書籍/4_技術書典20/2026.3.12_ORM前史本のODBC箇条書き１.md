## この章について
- 前章まででRDBの歴史と、Embedded SQLという初期のクエリーの仕組みを説明しました。Embedded SQLは当時のRDBとプログラミング言語とを結ぶ道を作り、多くのシステムにRDBを利用するための手立てを提供しました。ただ、その一方でプレコンパイルによる強いDBベンダー依存性やクエリが静的になるなど、課題も持っていました。
- この章で紹介するODBCは接続先DBを意識させない透過的なクエリ発行や動的なクエリ生成など、Embedded SQLの課題を解決するソリューションです。ODBCは現代でもエンタープライズ領域では利用されるツールであり、実務の中でお目にかかることもあるかもしれません。

## 第一節
- 第一節ではODBC登場までの歴史を扱います。ODBCが登場したのは1992年のことであり、Embedded SQLの登場した1982年から10年後のことでした。この時代はアメリカソフトウェア産業の趨勢がIBMからMicrosoftへ大きく動いた時代であり、奇しくもEmbedded SQLとODBCの開発元もその両者です。ではどのような時代背景が当時はあったのでしょうか？
- 当時
	- 圧倒的シェアをもっていたIBM
	- 1970年代のPCブーム、初めは組み立てが必要でありギークむけだったけど、1970年代半ばにはオールインワンのPCが発売される、それがAppleⅠ。そして1979年にApple2が出て、そこにBISICALCというキラーアプリが搭載される、これは表計算ソフト。一気に商業的な価値を持つ。なおこのときApple2にBASIC言語を提供していたのがMicrosoft
	- 遅れて1981年にIBMも参入し、Apple2を追い返した。このときIBMはPC用のOS制作をMicrosoftに依頼した。MSはこのDOSを急ピッチで作り、そして完全にIBMに渡すのではなく著作権を手元に残す形でライセンス提供した。それによって、MicrosoftはこのOS ー DOSを他社に販売する権利も持っていた。なのでIBMクローンの様々なPC全てにDOSが入った。
	- 当時のIBMとMicrosoftは蜜月関係であり、IBMとMicrosoftはIBM PC向けの独自OSであるOS/2を共同で開発しています。しかし、その裏でMicrosoftはGUI OSであるWindowsの開発に乗り出します。85年のWindows1.0, 90年のWindows3.0と開発を続け、ついに92年のWindows3.1では爆発的な売り上げを見せます。一方のOS/2はというと、1990年に開発の責任の大半をIBMが負うことが決まり、1991年にはMicrosoftは撤退しています。
	- そして、このWindows3.0からの時代で流行したのがクライアントサーバー方式です。それまでIBMが圧倒的なシェアを誇っていた汎用機中心のシステムから、WindowsなどGUIのクライアントがサーバーにアクセスする方式となりました。
	- そしてそのサーバー側に立っていたのがRDBでもあります。80年代を通してデータベース製品は多様化し、そして90年代に入る頃のエンタープライズシステムでは部署ごとに好きにDBを入れている、というような光景も一般的になりました。
- ここで時代のニーズを一度まとめて見ましょう
	- ひとつは、GUIによる豊かな情報表現です。特にExcelなどの表計算ソフトの登場がありました。かつてのCUIシステムと比較し、より表現力の高く動的なクエリが求められました。
	- 次に、クライアントサーバー方式の確立と、DBの乱立です。単一の企業でも複数のデータベース製品を導入していることがままあり、それらを横串に見るニーズが生まれています。
- CLI
	- これらの課題はEmbedded SQLでは解決できない問題でした。そういった状況を打破するために、1989年にCall Level Interfaceという規格が計画されます。
	- データベースベンダーがあつまり、仕様に沿って各々のライブラリ開発
- ODBC
	- これを実装したのがMicrosoft
	- 策定に関わり、1992年にODBC1.0を公開した。1994には2.0、1996にはMicrosoftがODBC3.0で確定させる
	- 企画としてもX/Open、ISOと認められてより上位の標準化団体で策定
	- これはCLI規格に則りながら、いくつかの機能も取り込んでいる
		- トランザクション
		- 透過的処理
- 現代
	- 現代では透過的接続のニーズが低くなっていることやその後のOSSブームなどもあり、ODBCは歴史あるエンタープライズや、複数DBを触らないといけない状況でしか使われない。言語ごとにドライバーが用意されていることが多い。それでもこの技術はすんごいね

## 第二節
- それでは第二節では仕様と使い方を見ていこう
	- 基本的には[[『SP 96 SQL Access Group's Call-Level Interface』要約]]を参照しながら進める
- 仕様ではこんなのが規定されてた

> - ハンドルの割り当てと解放（8呼び出し）
> - 属性の取得と設定（10呼び出し）
> - データベース接続のオープンとクローズ（2呼び出し）
> - デスクリプタへのアクセス（6呼び出し）
> - SQL文の実行（9呼び出し）
> - 結果の取得（8呼び出し）
> - スキーマメタデータへのアクセス（4呼び出し）
> - イントロスペクション（内省）の実行（4呼び出し）
> - トランザクションの制御（2呼び出し）
> - 診断情報へのアクセス（3呼び出し）
> - 関数のキャンセル（1呼び出し）

- 使い方はこんなの。まずコネクションして、、とか


```c
/* 環境ハンドルを割り当てる */
SQLAllocHandle(SQL_HANDLE_ENV, SQL_NULL_HANDLE, &henv);
/* 接続ハンドルを割り当てる */
SQLAllocHandle(SQL_HANDLE_DBC, henv, &hdbc);
/* データベースに接続する */
if (SQLConnect(hdbc, server_name, SQL_NTS, uid, SQL_NTS, pwd, SQL_NTS) !=
         SQL_SUCCESS)
    return(PrintErr(SQL_HANDLE_DBC, hdbc));
/* 文ハンドルを割り当てる */
SQLAllocHandle(SQL_HANDLE_STMT, hdbc, &hstmt);
/* SQL文を実行する */
if (SQLExecDirect(hstmt, sqlstr, SQL_NTS) != SQL_SUCCESS)
    return(PrintErr(SQL_HANDLE_STMT, hstmt));
/* 文の種類を確認する */
SQLGetDiagField(SQL_HANDLE_STMT, hstmt, 0, SQL_DIAG_DYNAMIC_FUNCTION_CODE, 
        (SQLPOINTER)&stmttype, 0, (SQLSMALLINT *)NULL);
/* SQL文を処理する */
DoStatement(stmttype);
/* 文ハンドルを解放する */
SQLFreeHandle(SQL_HANDLE_STMT, hstmt);
/* データベースから切断する */
SQLDisconnect(hdbc);
/* 接続ハンドルを解放する */
SQLFreeHandle(SQL_HANDLE_DBC, hdbc);
/* 環境ハンドルを解放する */
SQLFreeHandle(SQL_HANDLE_ENV, henv);
```
