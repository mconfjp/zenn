## 未調査（優先度高）
- [ ] ⑰ Embedded SQL → CLI → ODBC の3段階の進化（★★★）
	- コアアイデア: DB接続の抽象化は3段階で進化した。それぞれが「前の方式の何が辛かったか」への回答
	- 要調査: CLI時代のベンダー固有APIの違い（OracleのOCIとDB2 CLIの具体的なコード差）
	- **調査メモ（2026.3.1）**: Embedded SQL → CLIの移行動機は「クライアント・サーバーアーキテクチャへの移行」と「動的クエリの必要性」。SAGは1989年設立、1990年にCLI開発開始（出典: [[『SP 96 SQL Access Group's Call-Level Interface』本文]]）。Embedded SQLの実行計画はRebindしないと古いまま——これも運用上の欠点として記録されている（出典: IBM DB2 docs）
- [ ] W1. WIPノート補完「1980年代にORMが生まれなかった背景」（★★★）
	- 対象: `WIP_1980年代には現代のORMほどのデータベース抽象化は行われなかったが、それは技術的文化的な背景があった.md`
	- 「なに」セクションを書いてWIPを完成させる
- [x] E1. Embedded SQL の技術的な説明（★★★）
	- コアアイデア: EmbeddedSQLはホスト言語（COBOL・C等）の中にSQL文を埋め込む方式。プリコンパイラが前処理する
	- 要調査: 具体的なコード例、プリコンパイラの仕組み、当時の利用文脈
- [ ] E2. CLIはどうやってEmbedded SQLより簡潔さを実現したか（★★★）
	- コアアイデア: Embedded SQLはビルド時にCOBOL側とDB2側の両方の準備が必要で重たかった。CLIはどうやってそれを回避したか
	- 要調査: CLIのAPI呼び出しの仕組み、Embedded SQLとの設計上の違い、コンパイル型言語前提の制約をどう解消したか
	- **調査メモ（2026.3.1）**: Microsoft ODBC公式ドキュメントに「ユーザーが自分でクエリを入力できるアプリはEmbedded SQLでは実現不可能だった（SQL statements cannot be hard-coded）」と明記されている。CLIが解決した問題の軸は2つ：①動的クエリの実現、②ベンダー移植性（別軸）。参照: https://learn.microsoft.com/en-us/sql/odbc/reference/embedded-sql
- [ ] O1. ODBCがしなかったこと（オブジェクト変換の不在）（★★★）
	- コアアイデア: ODBCは接続の抽象化はしたが、結果セット（行・列）をオブジェクトに変換する仕組みは提供しなかった。「接続は解決、変換は未解決」というセクション0の結論にあたる
	- 要調査: ODBCが返すデータ形式（行・列のカーソル等）、アプリ側でどう手作業マッピングしていたか
