### ① SQL Access GroupとCLIによる標準化の試み（1989年）

**コアアイデア**

**ネタ**

- ODBC（Open Database Connectivity）：1992年にMicrosoftが標準化。CILの考え方を実装した
- SQL Access Group：標準化を推進した業界団体（詳細調査が必要）
- Embedded SQLとの違い：Embedded SQLはソースコードにSQL直書き。CLIは実行時にSQL文字列を渡す

## セクション1: インピーダンスミスマッチ

### ⑤ オブジェクト同一性とは何か（メモリアドレス vs 主キー）

**重複確認の結果**
`インピーダンスミスマッチには「粒度」「サブタイプ」「オブジェクト同一性」の問題がある.md` には3つの問題をざっくり列挙しているだけ。オブジェクト同一性そのものを掘り下げたノートはないため、独立したpermanentとして作れる。

**コアアイデア**
オブジェクトの「同一性」とRDBの「一意性」は根本的に異なる概念で、これがO-Rマッピングの最も本質的な摩擦の一つ。

**ネタ**
- オブジェクトシステムの同一性：メモリアドレスに基づく暗黙の同一性。同じ状態の2つのオブジェクトは別々のオブジェクト
- リレーショナルシステムの同一性：状態に暗黙的。まったく同じ状態の2行はデータ破損とみなされる
- 解決策：リレーショナルシステムに人工的な一意性（サロゲートキー、自動インクリメント整数列）を持たせる
- Javaの場合：`==`（参照同一性）と`equals()`（値等値性）の二重の概念
- DDDの例外：DDDのエンティティは意図的に識別子（ID）を持つ設計

### ⑥ 継承のマッピング3戦略（Single/Class/Concrete Table）

**コアアイデア**
オブジェクトの継承をRDBで表現しようとすると、どの戦略を選んでも何かを犠牲にしなければならない。

**ネタ**
- **Class Table Inheritance（クラスごとのテーブル）**
  - 各クラスが独自のテーブルを持つ
  - 問題：多数の高価なJOINが発生する
- **Concrete Table Inheritance（具象クラスごとのテーブル）**
  - 最派生クラスのみにテーブルを作成（中間クラスはテーブルなし）
  - 問題：重大な非正規化コスト、スーパークラスの変更が全テーブルに波及
- **Single Table Inheritance（クラスファミリーごとに単一テーブル）**
  - 継承階層全体で単一のテーブル。typeカラムで区別
  - 問題：多くのNULL列が発生、NULL可能性制約が失われる
- Fowler『PoEAA』でパターン名が定義された

### ⑦ OODBが失敗した3つの理由

**コアアイデア**

### ⑧ OODB vs ODBC vs ORMという3つのアプローチのトレードオフ

**コアアイデア**
インピーダンスミスマッチへの解法として、「DB側を変える（OODB）」「接続を標準化する（ODBC）」「変換層を作る（ORM）」という3つの方向があった。

**ネタ**
- **OODB（DBを変える）**：インピーダンスミスマッチが発生しない。しかし既存RDB資産を捨てる必要がある
- **ODBC/JDBC（接続を標準化する）**：DBとの会話方法を共通化。変換は開発者の責任
- **ORM（変換層を作る）**：RDBを維持したまま、変換を自動化。ただしTed Newardが「ベトナム」と呼んだ複雑性がある
- 3つが同時代に競合していた（1990年代）
- 結果的にORMが主流になったのは、RDB側の資産・スキル・標準が既に巨大だったから

## セクション3: ORMの隆盛

### ⑬ Active Record vs Data Mapperパターンの本質的な違い

**コアアイデア**
ORMの設計思想は大きく2つに分かれる。Active RecordはドメインオブジェクトとDBアクセスを一体化し、Data Mapperは分離する。

**ネタ**
- Fowler『Patterns of Enterprise Application Architecture』（PoEAA）で両パターンが命名・定義された
- **Active Recordパターン**：
  - オブジェクト自体がDBアクセスの責任を持つ（`User.save()`, `User.find()`）
  - シンプルなCRUDに向いている
  - Ruby on Railsが採用（2004年〜）
- **Data Mapperパターン**：
  - オブジェクトとDB操作を分離。Mapperが変換を担う
  - ドメインロジックの独立性が高い
  - TopLink（1994年〜）、Hibernate（2002年〜）が採用
- 設計思想の違い：ドメインモデルの独立性をどこまで重視するか

### ⑮ JPAの標準化とHibernateの関係

**コアアイデア**
HibernateはJava標準のJPA策定に直接関与した。OSSが事実上の標準を作り、そこから公式標準が生まれた逆転現象。

**ネタ**
- JPA（Java Persistence API）：Javaの永続化標準仕様
- JPA 1.0：JSR-220として2006年にリリース（EJB 3.0の一部）
- Gavin King（Hibernate作者）がJPA仕様策定グループに参加
- HibernateはJPA 1.0の参照実装の一つとなった
- 経緯：EJBのEntity Bean（JSR-153）が複雑すぎると批判 → EJB 3.0でJPAを導入
- 「Java Persistence APIがEJB Entity BeansとJDOの両方の代替として受け入れられた」（Neward）

### ⑯ MySQLのInnoDBによるエンタープライズ化

**コアアイデア**
MySQLにトランザクション対応のInnoDBが統合されたことで、OSSのRDBがエンタープライズ用途に耐えるようになり、ORMの普及基盤が整った。

**ネタ**

- 2001年頃：MySQL 3.x にInnoDB統合が始まる
- 2003年：MySQL 4.0 でInnoDB安定版が標準同梱
- それ以前のMySQL：MyISAMのみ → トランザクションなし、外部キーなし
- ORM普及との関係：トランザクション・外部キーがないとORMの恩恵が限定的。InnoDBがあってこそ
- 2005年：OracleがInnobase Oyを買収（MySQLとは別で）

### ⑰ Embedded SQL → CLI → ODBCの3段階の進化

**コアアイデア**
DB接続の抽象化は3段階で進化した。それぞれが「前の方式の何が辛かったか」への回答になっている。

**ネタ**
- Embedded SQL（1970年代後半）：ソースコードにSQL直書き。プリコンパイラが必要。DBスキーマへの依存が強い
- CLI（1980年代後半）：実行時に関数でSQLを渡す。プリコンパイル不要。ただしDBベンダーごとに関数名・呼び出し方が違った（OracleはOCI、DB2はDB2 CLIなど）
- ODBC（1992年）：CLIの発想をMicrosoftが標準化。`SQLConnect()` `SQLExecDirect()` などどのDBでも同じAPIで呼べる。裏でDriverがDB差異を吸収

**参照元資料**
- [SP 96: SQL Access Group's Call-Level Interface（Dr. Dobb's, 1996）](https://jacobfilipp.com/DrDobbs/articles/DDJ/1996/9613/9613a/9613a.htm)
- [SQL Access Group - Wikipedia](https://en.wikipedia.org/wiki/SQL_Access_Group)

**既存permanentへのリンク候補**
- `[[ORMに先行するDB抽象化は存在する]]`

**要調査**
- CLI時代のコード例（OracleのOCIとDB2 CLIの具体的な関数名の違い）
- ODBC以前にベンダー固有APIを使っていたときの実務上の苦労

### ⑱ 接続の標準化（ODBC）はORMの必要条件だったのか？

**コアアイデア**
ODBCなしにORMを作ると、DB製品ごとに接続ドライバを自前でメンテしないといけない。ODBCの登場（1992年）とTopLink for Smalltalk（1994年）の時系列は偶然ではないかもしれない。

**ネタ**
- ODBC登場：1992年
- TopLink for Smalltalk：1994年
- 現代の類比：TerraformはクラウドプロバイダごとにProviderを自前でメンテしている
- 仮説：標準化がなければORMはDB製品ごとの対応コストが高すぎて普及しなかった

**参照元資料**
- なし（仮説）

### ⑲ TopLinkを作らせたニーズは何か（1994年、Java以前）

**コアアイデア**
Webも未成熟、Javaもない1994年に、The Object PeopleがO-Rマッピングツールを作った動機は何か。技術的可能性ではなくニーズが先にあったはず。

**ネタ**
- 1994年：Javaはまだない。Webは黎明期。Smalltalkが主要OOP言語
- The Object People：カナダの会社。Smalltalk開発者コミュニティ
- 当時の業務アプリ：RDBとSmalltalkを組み合わせた企業システムが存在していたはず
- 仮説：毎回手書きしていた変換コードの辛さが限界に達した開発者がいた

### ⑳ 技術的可能性よりニーズが先、という法則のORM版

**コアアイデア**
ORMは技術的には1980年代に実現可能だったかもしれないが、作られなかった。それは「作れなかった」のではなく「作る必要がなかった」からではないか。ニーズがいつ・なぜ生まれたかがORMの本質的な歴史。

**ネタ**
- コンピュータ黎明期でも同様のパターンは多い（技術より需要が先）
- 「できる」と「やる」の間にあるのはニーズとコスト
- OOPの普及 → オブジェクトで世界を表現したくなる → RDBとの乖離が辛くなる → ORM登場、という順序

**参照元資料**
- なし（仮説）
