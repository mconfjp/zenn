>**コンセプト**: RDB（データを概念にした）とOOP（プログラムの抽象度を上げた）のインピーダンスミスマッチを解消するためにORMが登場した

- あー多分ここに至るまでに、OOPとRDBで思想をちゃんと全面に出していかないとダメなんだろうなあ

	- その違いをもっとこう、明確にここで示してあげたい
		- まあ難しくいうとインピーダンスミスマッチなんだろうけど、、しかしあんまり理解できてない、しっくりきてない

	- ORM前史
	- ふたつのパラダイム

### 0. ORM以前の橋渡し（1,000文字）
- Embedded SQL：プログラムの中にSQL直書き
- CLI / ODBC：接続方法の標準化。データの変換は手作業
- 「DBと話す方法」は提供したが「オブジェクトとテーブルの変換」はしなかった

- ここで書くこと
	- はじめに
		- OOPとRDBの歴史を書いて、その合わさったところのアプリケーション実務の話があって、じゃあ永続化レイヤー抽象化はどうなってたのか
		- 多分、これ、ファイル書き込みとか、RDB以前の抽象化もあったんだよな、、
		- 独立したコンポーネントになったんだ
			- COBOLが出てきて、データとロジックが分離したとかいう話
	- RDBにおける抽象化
		- EmbeddedSQL、ODBC
		- それぞれの年代と、それ以前の話も気になる
- わかること
	- EmbeddedSQL、ODBC
	- まだ接続の話だったということ
- わからんこと
	- それ以前の永続化レイヤー抽象化はどうなってた？　してなかったんだったらしてなかったことを明示しておきたい
	- COBOLによるデータとロジックの分離
	- RDB出てきてからORMが出るまで時間差があって、それまではどれくらいRDBとOOPの組み合わせがあってそれまでどうしてたのか
	- まあJavaないとね

### 2. ORMの登場（2,000文字）
	- 1994年 TopLink for Smalltalk：最初のORM、商用
	- 1996年 TopLink for Java
	- 2002年 Oracle買収 / Hibernate登場：商用→OSSへの転換
	- 2006年 ActiveRecord / Rails：フレームワーク標準装備へ
- わかること
	- 各種製品がいつ出たのか
	- 周辺の情報
		- MySQL4.0、Java
- わからんこと
	- 利用状況かなーーー
	- いつ頃完全に馴染んだ？
		- 1994に登場して、フレームワーク標準になったのはいつが初め？
	- TopLinkが最初どこまで機能を載せていたのか
### 3. ORMが当たり前になるまで（1,000文字）

	- TopLink→Hibernate→ActiveRecordの流れ自体がソフトウェア産業の変化

- わかること
	- ORMに付属的に求められたもの？
- わからないこと

### この章の意味
- ORMの歴史を書く
	- これは、永続化レイヤー抽象化の歴史の、かなり袋小路というか、ずっとこう発展しきったところ
	- RDBとOOPという前提の上で、どう抽象化がなされたのか
	- その前提なしだった場合、どうなるんだ
- データとロジックの確定
	- データは変わらず、ロジックは変化する
		- ロジックの変化を資本主義が求めてくる
		- 振る舞いとデータの整合性
	- インピーダンスミスマッチが起きる
	- 解決するためのORM

	- これが結局なんなんだろうか、、、
