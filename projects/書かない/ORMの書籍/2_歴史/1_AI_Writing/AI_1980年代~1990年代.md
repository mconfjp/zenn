# 1980年代~1990年代：GUIコンピュータの登場とOOPの商用利用

1980年代から1990年代は、1970年代に研究所で開発されたGUI技術とオブジェクト指向プログラミングが商用市場に本格的に展開され、現代的なソフトウェア開発環境の基盤が確立された時代である。この時期の技術革新は、後のORM技術が必要とする「オブジェクトとデータの分離」という概念的基盤を実用レベルで実現した。

## Xerox StarからMacintoshへ：GUIの商用化

1981年、Xeroxは研究所で開発されたAlto技術を商用化したXerox Star workstationを発売した。Starは世界初の商用GUIコンピュータとして、ビットマップディスプレイ、マウス操作、ウィンドウシステム、アイコンベースのファイル管理を統合した革新的なシステムであった[^1]。しかし、16,595ドルという高価格と限定的な用途により、商業的には失敗に終わった。

Starの技術的遺産は、1984年のApple Macintoshによって真に開花した。Macintoshは、Xerox PARCで開発されたGUI技術を一般消費者向けに再設計し、2,495ドルという相対的に手頃な価格で提供した[^2]。重要なのは、MacintoshのGUI設計がSmalltalkの影響を強く受けていたことである。

Steve Jobsは1979年にXerox PARCを訪問し、AltoとSmalltalkのデモンストレーションを見学した。この経験は、Macintoshの設計思想に決定的な影響を与えた[^3]。MacintoshのObject Pascal環境は、Smalltalkのオブジェクト指向概念を実用的なアプリケーション開発に適用した初期の成功例となった。

興味深いのは、これらのGUIシステムが一旦Smalltalkから離れて独自の実装で登場したことである。Xerox StarはMesa言語で、MacintoshはPascalとアセンブリ言語で実装されていた。しかし、その設計思想と操作概念は明確にSmalltalkのオブジェクト指向パラダイムを反映していた[^4]。

## IBMのSmalltalk移行プロジェクト：企業レベルでのOOP導入

1990年代初頭、IBMは企業アプリケーション開発において画期的な決断を下した。同社は大規模なSmalltalk移行プロジェクトを開始し、後にVisualAgeとして知られる統合開発環境を構築した[^5]。

この移行の背景には複数の要因があった。第一に、GUI アプリケーションへの需要が急速に高まっていた。従来のキャラクターベースのインターフェースでは、ユーザーの期待に応えることが困難になっていた。第二に、IBM社内では既にClassCという内部開発言語でオブジェクト指向プログラミングの実験が行われており、OOPの利点が実証されていた[^6]。

重要なのは、これが単なる採用言語の変更ではなく、開発環境全体の移行であったことである。VisualAge Smalltalkは、統合開発環境（IDE）、バージョン管理システム、チーム開発支援ツールを一体化した包括的なソリューションを提供した[^7]。この経験は、オブジェクト指向開発における「環境の重要性」を実証し、後のJava開発環境やモダンなIDEの設計に大きな影響を与えた。

IBMのSmalltalk移行プロジェクトは、エンタープライズレベルでのオブジェクト指向プログラミングの実用性を証明した最初の大規模事例となった。このプロジェクトで得られた知見は、後にJavaプラットフォームの企業向け機能設計に活用された[^8]。

## Windows 95とGUIの大衆化

1995年のMicrosoft Windows 95の登場は、GUIコンピューティングの決定的な転換点となった。Windows 95は、それまで主に専門家や愛好家に限られていたGUIを一般大衆に普及させた[^9]。

Windows 95の成功は、技術的革新だけでなく、マーケティング戦略の勝利でもあった。Microsoftは「Start me up」キャンペーンで3億ドルを投じ、GUIコンピューティングを日常的なツールとして位置づけた[^10]。この大衆化は、ソフトウェア開発業界に根本的な変化をもたらした。

GUI アプリケーションの需要急増により、従来の手続き型プログラミングの限界が明確になった。ウィンドウ、ボタン、メニューといったGUI要素を効率的に管理するには、オブジェクト指向プログラミングの概念が不可欠であった。Windows 95の普及は、オブジェクト指向プログラミングの商用利用を加速する重要な触媒となった[^11]。

## Javaの登場：プラットフォーム独立性の実現

1995年、Sun MicrosystemsはJavaプログラミング言語を発表した。Javaは「Write Once, Run Anywhere」（一度書けば、どこでも動く）というスローガンで、プラットフォーム独立性を実現した[^12]。

Javaの設計は、1970年代のUNIXが示した移植性の概念を、オブジェクト指向プログラミングの文脈で再実現したものであった。Java仮想マシン（JVM）は、異なるオペレーティングシステムとハードウェアプラットフォーム間でのコード互換性を提供した[^13]。

Javaの登場は、Smalltalkの衰退を決定づけた。Smalltalkは純粋なオブジェクト指向言語として技術的に優れていたが、C言語の構文に慣れた開発者にとってJavaの方が習得しやすかった。また、Javaは無料で利用でき、オープンな標準として普及した[^14]。

重要なのは、Javaがエンタープライズアプリケーション開発に特化した機能を提供したことである。JDBC（Java Database Connectivity）は、データベースアクセスの標準化を実現し、後のORM技術の基盤となった[^15]。

## オブジェクト指向とデータベースの接点

1980年代から1990年代にかけて、オブジェクト指向プログラミングの普及は、データベースアクセスにおける新しい課題を浮き彫りにした。従来のSQL中心のアプローチでは、オブジェクトの状態とリレーショナルデータベースのテーブル構造の間に「インピーダンスミスマッチ」が生じることが明らかになった[^16]。

この問題に対する初期のアプローチとして、オブジェクトデータベース管理システム（OODBMS）の研究が活発化した。1980年代後半から1990年代前半にかけて、Gemstone、Objectivity/DB、Versantなどの商用OODBMSが登場した[^17]。

しかし、OODBMSは既存のリレーショナルデータベースの置き換えとしては普及しなかった。企業の既存システムとの互換性、SQL の標準化、リレーショナルデータベースの成熟したエコシステムなどの要因により、OODBMSは特定の用途に限定された[^18]。

この経験は、オブジェクトとリレーショナルデータベースの「橋渡し」技術の必要性を明確にした。1990年代後半から2000年代初頭にかけて登場するORM技術は、この課題に対する実用的な解決策として発展することになる。

## 結論

1980年代から1990年代は、1970年代に研究所で開発された技術が商用市場で実用化され、現代的なソフトウェア開発環境の基盤が確立された時代である。Xerox StarからMacintosh、そしてWindows 95に至るGUIの普及は、オブジェクト指向プログラミングの必要性を実証した。

IBMのSmalltalk移行プロジェクトは、エンタープライズレベルでのOOP導入の可能性を示し、Javaの登場はプラットフォーム独立性とオブジェクト指向プログラミングを統合した。これらの発展は、オブジェクトとリレーショナルデータベースの統合という新しい技術的課題を生み出し、ORM技術登場の直接的な前提条件を形成した。

1990年代末までに、オブジェクト指向プログラミングは主流の開発手法となり、GUIアプリケーションは標準的なソフトウェア形態となった。この環境の中で、オブジェクトの状態を効率的に永続化する技術の必要性が高まり、1994年のTopLink for Smalltalkを皮切りとするORM技術の本格的な発展が始まることになる。
