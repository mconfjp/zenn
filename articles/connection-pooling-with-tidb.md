## コネクションプールのきほん

コネクションプールとは、アプリケーション側がデータベースへの接続をあらかじめ確保しておき、リクエストのたびに使い回す仕組みです。接続の確立はそれなりにコストがかかる処理なので、手元で確保しておくことにより通信にかかる時間を削減します。

プールの設定はアプリケーションの起動時に行います。主にORマッパーがここらへんをよしなにやってくれることが多く、「最大何本のコネクションを持つか（`max`）」と「コネクション取得を何秒待つか（タイムアウト）」を指定することが多いです。なおこの設定は ORM やデータベースライブラリごとに書き方が異なり、大変面白いです[^orm-pool-settings]。

[^orm-pool-settings]: 代表的な ORM の設定方式は大きく3パターンに分類でき、次のとおり。①URLクエリパラメータ方式：Prisma v6 以前で使われていた。データベースとの接続URLに`?connection_limit=10&pool_timeout=10`というような形でクエリパラメータを付与し、prisma側に接続URLが渡ったタイミングでこれを読み取り処理する形式。　②設定オブジェクト方式：SQLAlchemy・TypeORMなどで採用。初期化時に設定ファイルからオブジェクトで渡す。　③メソッドチェーン方式：GORMなどで採用。接続後に `db.DB()` 経由で `SetMaxOpenConns()` 等を呼ぶ。

## どんな問題が起きるか

### コネクションプールの枯渇

データベース側には、同時に受け付けられる接続数の上限があります。アプリケーション側がプールとして確保しようとするコネクション数の合計がこの上限に達すると、新たな接続が拒否されエラーになります。
コネクションプーリングにはアプリ側で上限を設けることでこれに抵触しないようにするという目的もあるのですが、相乗り環境とECSスケールアウトが相まると少し具合の悪いことになります。

### 相乗り環境との相性

相乗り環境では、複数のアプリケーションが同じデータベースクラスターに接続します。各アプリが `max=10` のプールを持っているとして、アプリが50本あれば単純計算で最大500コネクションを要求することになります。

### ECSスケールアウトとの相性

ECSではタスクが増えるたびに、そのタスクが新たにプールを張ります。1タスクあたり `max=10` であれば、タスクが10に増えると100コネクション、20に増えると200コネクションが必要になります。^[そんなにタスクが増えることはないけど。。]

相乗り環境にECSのスケールアウトが重なると、「アプリ数 × タスク数 × プール上限」という積が一気に膨らみ、データベースの接続上限に触れやすくなります。

## 本当に問題なのか？
では、これらの状況が合わさったとき、弊社環境では本当に問題になったのでしょうか？
答えは「問題なさそう」でした。調査の流れは以下の通りです。

-  TiDBのコネクション上限を確認する
- コネクション数を推測する
- コネクション数を計測する

###  TiDBのコネクション上限を確認する
上限は以下のコマンドで確認することができます。

```sql
SHOW VARIABLES LIKE 'max_connections';
```

### コネクション数を推測する
マイクロサービス数 * タスク数 * コネクション数で計算をしました。
コネクション数はTypeORMであればデフォルトで10, Prismaでも10程度だったため、単純な計算で割り出せました。

### コネクション数を計測する
また、そもそもコネクション数は見えます^[上の計算をやってる時に気がついた]。TiDB CloudであればTiDB Cloud Clinicのメトリクスにて、`TiDB-Cluster-Clinic > Server > Connection Count`で実測値を知ることができます。

結果として、コネクション上限の20%程度が普段の接続数と分かり、まあこれなら大丈夫でしょうとなりました。





## こう対応する

対策は大きく2種類あります。

### ① アプリケーション側でコネクション数を管理する

データベースの接続上限を超えないよう、各アプリのプール設定を調整する方法です。おおよそ次の流れで進めます。

1. **データベース側の上限を確認する**：TiDB Cloud であればコンソールやドキュメントで最大接続数を確認します
2. **アプリケーションごとの使用量を見積もる**：アプリ数・タスク最大数・1タスクあたりのプール上限から、合計コネクション数を計算します
3. **各アプリのプール上限を調整する**：合計が上限を超えないように `max` の値を設定します

計算式の目安は次のとおりです。

```
1タスクあたりの max ≦ DB接続上限 ÷ アプリ数 ÷ タスク最大数
```

シンプルですが、アプリや環境が増えるたびに再計算が必要になるため、管理コストが上がりやすいというデメリットがあります。

### ② コネクションプーリングプロキシを導入する

アプリケーションとデータベースの間にプロキシを挟み、プロキシ側でコネクションを一元管理する方法です。各アプリはプロキシに接続し、プロキシがデータベースへの接続をまとめて管理します。

アプリが増えても設定を個別に変更する必要がなく、スケールアウト時の接続数増加も吸収しやすいのが利点です。ただし、プロキシ自体の運用が増えるというトレードオフがあります。

## まとめ

相乗り環境 × ECS のスケールアウトという構成では、コネクション数が予想以上に積み上がりやすいため、あらかじめ上限を意識した設計が必要です。

なお、コネクション数を増やしすぎるとデータベース側の負荷も上がります。接続を確立・維持すること自体がDBのリソースを消費するため、「上限に達していないから安全」ではなく、適切な範囲に収めることを意識してください。
