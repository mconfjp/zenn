## コネクションプールの基本

コネクションプールとは、アプリケーション側がデータベースへの接続をあらかじめ確保しておき、リクエストのたびに使い回す仕組みです。接続の確立はそれなりにコストがかかる処理なので、手元で確保しておくことにより通信にかかる時間を削減します。

プールの設定はアプリケーションの起動時に行います。主にORマッパーがここらへんをよしなにやってくれることが多く、「最大何本のコネクションを持つか（`max`）」と「コネクション取得を何秒待つか（タイムアウト）」を指定することが多いです。なおこの設定は ORM やデータベースライブラリごとに書き方が異なり、大変面白いです[^orm-pool-settings]。

[^orm-pool-settings]: 代表的な ORM の設定方式は大きく3パターンに分類でき、次のとおり。①URLクエリパラメータ方式：Prisma v6 以前で使われていた。データベースとの接続URLに`?connection_limit=10&pool_timeout=10`というような形でクエリパラメータを付与し、prisma側に接続URLが渡ったタイミングでこれを読み取り処理する形式。　②設定オブジェクト方式：SQLAlchemy・TypeORMなどで採用。初期化時に設定ファイルからオブジェクトで渡す。　③メソッドチェーン方式：GORMなどで採用。接続後に `db.DB()` 経由で `SetMaxOpenConns()` 等を呼ぶ。

## どんな問題が起きるか

### コネクションプールの枯渇

データベース側には、同時に受け付けられる接続数の上限があります。アプリケーション側がリクエストを受け付けてデータベースに接続を試みコネクション数の合計がこの上限に達すると、新たな接続が拒否されエラーになります。
コネクションプーリングにはアプリ側で上限を設けることでこの接続上限に抵触しないようにするという目的もあるのですが、相乗り環境とECSスケールアウトが相まると少し具合の悪いことになります。

### 相乗り環境との相性

相乗り環境では、複数のアプリケーションが同じデータベースクラスターに接続します。各アプリが `max=10` のプールを持っているとして、アプリが50本あれば単純計算で最大500コネクションを要求することになります。

### ECSスケールアウトとの相性

ECSではタスクが増えるたびに、そのタスクが新たにプールを張ります。1タスクあたり `max=10` であれば、タスクが10に増えると100コネクション、20に増えると200コネクションが必要になります。^[そんなにタスクが増えることはないけど。。]

相乗り環境にECSのスケールアウトが重なると、「アプリ数 × タスク数 × プール上限」という積が一気に膨らみ、データベースの接続上限に触れやすくなります。

## 本当に問題なのか？
では、これらの状況が合わさったとき、弊社環境では本当に問題になったのでしょうか？　答えは「実測を見る限り問題なさそうだが、理論上は問題あり」でした。歯切れ悪いですね〜〜〜〜けど面白いので、以下の通り調査の流れを解説します。

-  TiDBのコネクション上限を確認する
- 実測：コネクション数を計測する
- 理論：コネクション数を推測する

###  TiDBのコネクション上限を確認する
上限は以下のコマンドで確認することができます。

```sql
SHOW VARIABLES LIKE 'max_connections';
```

弊社環境の場合、4000でした。多いねえ。
なおTiDBのデフォルト値は`0`で、*無限<infinite>*を意味します。

### 実測：コネクション数を計測する
続いて計測です。TiDB CloudであればTiDB Cloud Clinicのメトリクスにて、`TiDB-Cluster-Clinic > Server > Connection Count`で実測値を知ることができます。
みてみると、接続数は700弱とかなり余裕でした。ニッコリ。

### 理論：コネクション数を推測する
実測値で安全そうとわかったので、次は理論上どこまでいきそうか？　というのを計算してみます。これは`全体のタスク数 * コネクション数`で計算をしました。
AWSのECSクラスター一覧画面では各クラスターに幾つのタスクが存在するかみることができます。確認したところタスク数の合計は131とわかりました。
続いては１タスクあたりのコネクション数ですが、弊社が主に使っているORマッパーはPrismaとTypeORMであり、どちらもコネクション数のデフォルト上限は10です。そのため理論上のコネクション数は最大で`1310`とわかります。
と言いたかったのですが...

### PHP-FPMはコネクションをプールしない
弊社の歴史的なシステムはPHPで動いています。そしてPHPのプロセスマネージャであるPHP-FPMはリクエストのたびにプロセスをリセットするため、コネクションをプールできません。
じゃあPHPではコネクションを取り放題ってこと...？　というわけで、対策を考えてみました。

## こう対応する

対策はいくつか種類があります。考えた上で廃案にしたものも含めて紹介します。

1. 非推奨：PHP-FPMのワーカー数を制限する
2. 非推奨：TiDB側でDBユーザーに対するコネクション数制限をつける
3. 推奨：TiDBのResource Unitで制限する
4. 推奨：プロキシを入れる

### 非推奨：①PHP-FPMのワーカー数を制限する
大量のリクエストを受け付けた時にPHP-FPMが無限増殖するのが悪いわけなので、PHP-FPMのワーカー数自体を制限しよう、という考え方です。
ただ、これをやると単純に受け入れたいアクセスも受け入れられなくなることになります。

### 非推奨：②TiDB側でDBユーザーに対するコネクション数上限をつける
MySQLではDBユーザーごとにコネクション数上限を設けることが可能です^[https://dev.mysql.com/doc/refman/8.0/ja/user-resources.html] 。TiDBはMySQL互換のため、同様のことが可能であろうと考えました。
しかし、以下のドキュメントによるとTiDBではできないようでした。
https://docs.pingcap.com/ja/tidb/stable/security-compatibility-with-mysql/

### 推奨：③TiDBのResource Unitで制限する
DBユーザーごとのコネクション上限はつけられませんでしたが、同様の仕組みとして、Resource Unitでの制限があります。
Resource UnitとはTiDB上のリソース消費を総合的かつ抽象的に表現したもので、コネクション数やTiKVのCPU使用率、その他指標などをまぜこぜにしたものです。こちらをある程度制限することで、コネクション数を守ることができます。
ただ一点注意としては、Resource Unitは実リソースを無視した設定をすることが可能なため、甘い制限をかけてリソースが枯渇する、ということがないようにする必要があります。

### 推奨④： コネクションプーリングプロキシを導入する

アプリケーションとデータベースの間にプロキシを挟み、プロキシ側でコネクションを一元管理する方法です。各アプリはプロキシに接続し、プロキシがデータベースへの接続をまとめて管理します。

アプリが増えても設定を個別に変更する必要がなく、スケールアウト時の接続数増加も吸収しやすいのが利点です。ただし、プロキシ自体の運用が増えるというトレードオフがあります。

## まとめ

今回は「ECS × TiDB Cloud相乗り環境におけるコネクションプール枯渇問題」について考えてみました。コネクションプールって普段適当にしか考えてないんですが、改めて考えてみると面白かったです。

なお、コネクション数を増やしすぎるとデータベース側の負荷も上がります。接続を確立・維持すること自体がDBのリソースを消費するため、「上限に達していないから安全」ではなく、適切な範囲に収めることを意識する必要もありそうです。
