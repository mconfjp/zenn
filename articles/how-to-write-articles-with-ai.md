# どう書いてるか？
## Zettelkastenでのメモ

まずはじめに、Zettelkastenでメモを取っていきます。
大切なのはまだこれは散発的にメモをとっているだけで、記事の企画は特にたてていないことです。日々読書したり考えごとをした時に、こまめにメモを取ります。
例えば今回だと、以下のようなメモをとっていました。

```text
挫折経験の解釈は伴走者の存在による

# なに
挫折経験を失敗と捉えるか、成長機会と捉えるかは伴走者の存在、あなたのおかげがあるか

# 整理
- 挫折経験が単なる失敗の記憶になるか、それとも成長の機会になるのかは解釈
	- その解釈を決めるのは、その辛い経験の時に孤独を感じていたかどうか。伴走者がいれば、だれかに助けてもらえていたら前向きに捉えられる
	- 助けてもらえたから頑張れた、となると他人に過剰な努力を強制せずに済む
- [[解釈は選択可能であり、思わせた側には責任がない]]
	- 助けてもらえていたのに誰も助けてくれなかったと言い出す人もいるし、勝手に他人を師匠扱いして助けられた！　と思ってるやつもいる
```

## マッピングと追加調査
記事にするとは決めましたが、この時点ではまだ関連するメモがいくつかあるだけです。
ただ、ここまで来ると、メモを並べて眺めたときに「知識の地図」みたいなものが作れます。
どういうことかというと、例えば今回題材にしたメモたちがあったとき、

![](https://storage.googleapis.com/zenn-user-upload/c380a405f1a2-20250525.jpeg)
*手書きで書くことが多い*

これは図に起こすとこうなります。箇条書きで構造化するだけとかでも十分いいです。
このマッピング作業をする意図は、知識の空白地帯を探すことです。上の写真だと赤いペンでくちゃくちゃ〜〜っと書いてある箇所が本来論ずるべきだけどまだ言及されていない箇所です。
今回の記事では、坂井さんの記事を参考にしたり、関連する心理学の概念を調べたりしました。

## 文体ファイル
こんな感じのやつです。^[わかったような口をきいてくるのがしゃらくさい]
このファイル自体は過去に書いた記事を読み込ませて文体としてまとめてくださいと言ったら作ってくれました。導入とまとめについては文体の把握がうまくいかなくて、毎回自分の避けているテイストで書かれていたため、個別に文体をまとめてもらいました。

```text
# もりたの文体と技術的特徴

## 技術的な専門性と傾向

もりたさんはMySQLを中心としたデータベース技術に強い専門性を持つバックエンドエンジニアです。特に以下の分野に関する深い知識と経験が見られます：

- **MySQL/Aurora**: バージョンアップデート、パフォーマンスチューニング、内部構造の理解
- **SQL**: 複雑なクエリ（サブクエリ、ウィンドウ関数など）の理解と解説
- **データベース設計**: 外部キー制約、パーティショニング、正規化などの設計手法
- **データベース理論**: トランザクション、ACID特性、クエリ実行計画などの理論的背景

また、技術書の読書量が多く、様々な書籍から得た知識を整理して伝える能力に長けています。技術的な内容を深く掘り下げながらも、実務での応用を意識した記事が多いことが特徴です。

## 文体の特徴

もりたさんの文体には以下のような特徴があります：

1. **親しみやすさと専門性のバランス**：
   - 「ども、レバテック開発部のもりたです」という挨拶から始まることが多い
   - 砕けた表現と専門的な説明を織り交ぜる
   - 「〜〜」や「！！」などの記号を用いて親しみやすさを演出

2. **構造化された説明**：
   - 記事の冒頭で「これはなに」という章を設け、記事の目的と範囲を明確に示す
   - 見出しを効果的に使い、階層構造を明確にする
   - 箇条書きやテーブルを多用して情報を整理する

3. **ユーモアの活用**：
   - 冗談や軽いジョークを交えて硬くなりがちな技術的内容を和らげる
   - 自虐的な表現や、自身の失敗談を交えることもある
   - 「マジで顔から火が出るかと思った」など、感情表現が豊か

## 導入部の文体

導入部では以下のような特徴を持つ文体が望ましいです：

1. **簡潔な挨拶と自己紹介**：
   - 「ども、レバテック開発部のもりたです」という定型の挨拶
   - 必要に応じて簡単な近況を添える（「花粉症の薬を飲み始めました」など）

2. **記事の目的と範囲の明示**：
   - 何について書くのかを端的に述べる
   - 「この記事では〜について解説します」のような明確な宣言
   - 必要に応じて前提知識や対象読者を示す

3. **避けるべき表現**：
   - 「〜ではないでしょうか？」「〜と思いませんか？」などの読者への問いかけ
   - 「一緒に考えていきましょう」などの読者を巻き込む表現
   - 過剰な共感を求める表現

4. **好ましい表現**：
   - 「〜をご紹介します」「〜を解説します」など、提供する内容を明示する表現
   - 記事の背景や執筆動機を簡潔に述べる
   - 必要に応じて記事の構成を示す

## まとめ部の文体

まとめ部では以下のような特徴を持つ文体が望ましいです：

1. **記事内容の簡潔な総括**：
   - 「今回は〜をまとめました」のように記事の内容を端的に振り返る
   - 重要なポイントを簡潔に再確認する

2. **個人的な感想や経験の共有**：
   - 「〜で楽しかったです」など、素直な感想を述べる
   - 「〜と思いました」など、自分の考えを示す
   - 実践から得た気づきや学びを共有する

3. **避けるべき表現**：
   - 「〜してみませんか？」などの読者への呼びかけ
   - 「あなたも試してみてください」などの行動を促す表現
   - 「いかがでしたか？」などの問いかけ

4. **好ましい表現**：
   - 「以上、〜のご紹介でした」など、記事の終わりを示す簡潔な表現
   - 「〜が皆さんの参考になれば幸いです」など、控えめな期待の表明
   - 必要に応じて今後の展望や関連トピックへの言及

## 特徴的な表現

もりたさんの記事に頻出する特徴的な表現には以下のようなものがあります：

- **挨拶**: 「ども、レバテック開発部のもりたです」
- **感嘆**: 「〜〜！！」（波ダッシュと複数の感嘆符）
- **強調**: 「めちゃくちゃ」「だいぶ」「かなり」などの程度副詞
- **口語的表現**: 「〜だったりする」「〜みたいな」「〜っぽい」
- **専門用語の噛み砕き**: 「〜というのは、〜ということです」
- **補足情報**: 脚注や「:::message」ブロックでの補足説明

## 記事の基本構成

もりたさんの記事は一般的に以下のような構成をとっています：

1. **タイトル**: 具体的で内容を端的に表すもの
2. **導入部**:
   - 「これはなに」という見出し
   - 挨拶と自己紹介
   - 記事の目的と範囲の説明
   - 必要に応じて記事の構成や前提条件の説明
3. **本文**:
   - 大見出し、中見出し、小見出しを使った階層構造
   - 概念説明→具体例→実践的なアドバイスという流れ
   - コードブロックや図表を用いた視覚的な説明
4. **まとめ**:
   - 「おわりに」という見出し
   - 記事のポイントの再確認
   - 個人的な感想や学びの共有
   - 発展的な内容や関連情報への言及
5. **参考情報**:
   - 参考にした書籍やWeb記事のリスト
   - 関連する自身の記事へのリンク
```

```text
# これはなに

こんにちは。仕事をしていると、ちょっとした失敗で気持ちが落ち込み、その後の生産性が下がってしまうことはありませんか？

「あのタスクでミスしてしまった...」

「自分の提案が却下された...」

こういった出来事があると、数時間、場合によっては数日間も気持ちが沈んでしまい、本来の力を発揮できなくなることがあります。これは仕事の効率性だけでなく、自分自身の成長にも大きな障害となります。

# 前提の理論

## 坂井風太さんの理論

挫折経験は、人を成長させることもあれば、傷つけることもあります。どちらになるかを分けるのは、その出来事をどう「解釈」するかによります。

例えば、同じ失敗体験でも：
- 「あんな修羅場でも頑張れたんだから、また頑張れる」という解釈をする人もいれば
- 「前もダメだったから今回もダメに決まってる」と解釈する人もいます

この解釈の違いは、周囲の支援と解釈の助けによって生まれます。

- **実際に孤独にしないこと**：「一緒にやりましょう」と伴走してくれる人の存在
- **出来事の解釈を与えてくれること**：「ここはできていた、ここは伸び代」など、別の視点を提供してくれる人の存在

放っておくと、人間は「失敗が傷つきになる道」を歩みがちです。これを防ぐには意識的な取り組みが必要なのです。

## 認知行動療法の視点

認知行動療法では、私たちの感情や行動は「出来事→解釈→行動」というフローで決まると考えます。

例えば：
- 人がこちらを見て笑った→「俺は笑われている」→落ち込む
- 人がこちらを見て笑った→「楽しいことがあったんだろう」→いい気分

同じ出来事でも、解釈が変われば感情や行動も変わります。そして重要なのは、**解釈は自分で変えられる**ということです。出来事と解釈を分離して考えることで、感情や行動をコントロールできるようになります。

# 失敗を挫折にしてしまう思考のフロー

失敗が単なる傷つき体験になってしまうプロセスには、一定のパターンがあります。

1. **過剰な期待**：「絶対にうまくいく」「完璧にやり遂げる」という非現実的な期待
2. **期待を下回る出来事**：予想外の障害や問題の発生
3. **失敗であるという解釈**：「これは失敗だ」という評価
4. **周囲に迷惑をかけたという解釈**：「みんなに迷惑をかけた」「信頼を失った」という思い込み
5. **落ち込みと闇落ち**：自己評価の低下、無力感
6. **生産性の低下**：モチベーションの喪失、集中力の低下
7. **解釈の固着、次の失敗を呼び込む行動**：「やっぱり自分はダメだ」という思い込みが強化され、次の挑戦も避けるようになる

このフローのどこかに介入することで、失敗を挫折ではなく成長の機会に変えることができます。

# 対策

## リスクマネジメントによる過剰な期待の補正

失敗を防ぐ最初のステップは、非現実的な期待を持たないことです。

- プロジェクト開始時に起こりうるリスクを洗い出す
- 「うまくいかない可能性」も含めて計画を立てる
- 完璧主義を手放し、「十分に良い」状態を定義する

これにより、予期せぬ出来事が起きても「想定内」として受け止められるようになります。

## 下振れる出来事に「失敗」という評価をしない

期待通りにいかなかったことを即座に「失敗」と評価するのではなく、別の視点で見ることも可能です。

- 「失敗」ではなく「学びの機会」と捉える
- 「うまくいかなかった部分」と「うまくいった部分」を分けて考える
- 「何が起きたか」という事実と「それをどう評価するか」を分離する

例えば、プロジェクトが遅延したとしても「スケジュールの見積もりについて学べた」「リスクを早期に検知できた」という側面に目を向けることができます。

## チームビルディングで「自分ごと化」せず「私達ごと化」する

失敗を個人の問題ではなく、チーム全体の課題として捉えることで、孤独感を減らすことができます。

- 「私が失敗した」ではなく「私たちの課題だ」という視点を持つ
- チームメンバーと率直に状況を共有する
- 一人で抱え込まず、助けを求める

「この苦境に共に立ち向かっている」という解釈ができると、失敗体験も共有の成長機会になります。

## 部屋に灯りをつける - 解釈に怯えず、周囲の反応を確認する

失敗した後、「みんなから嫌われている」「信頼を失った」と思い込みがちですが、実際はそうでないことが多いものです。

- 勇気を出して周囲の反応を直接確認する
- 「どう思われているか」を想像で膨らませるのではなく、事実を集める
- フィードバックを求め、客観的な評価を得る

暗闇の中で怖い形に見えていたものが、灯りをつけてみれば単なる洋服掛けだったという経験は誰にでもあるはずです。同様に、自分の解釈に怯えるのではなく、実際の状況を確認することで不安は軽減されます。

## 成人発達理論の第三段階を超えて、他人軸から抜け出す

成人発達理論によれば、多くの人は「他者の評価」を重視する段階から、「自分の価値観」で判断する段階へと成長します。

- 「周りからどう思われるか」より「自分はどう思うか」を大切にする
- 自分の価値観や判断基準を明確にする
- 失敗を「評価の対象」ではなく「学びの過程」と捉える

他人の目を気にしすぎると、小さな失敗も大きな挫折に感じてしまいます。自分の成長という視点で物事を見られるようになると、失敗への耐性が高まります。

## プライドで持ち直す

時には、単純に「プライド」が助けになることもあります。

- 「こんなことで負けるわけにはいかない」という気持ち
- 「次は絶対にうまくやってみせる」という決意
- 「自分の本当の実力はこんなものではない」という自負

プライドは時に邪魔になることもありますが、挫折から立ち直る原動力にもなります。上手に活用しましょう。
```
