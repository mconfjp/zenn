## ロールアサインとは
### 概要
ロールアサインとは、チームで提供するべきサービスと役割を規定し、メンバーを割り振ることで責任の所在を明確にしたり、足りていない役割を把握してチーム運営に生かす取り組みです。

例えば「事業施策に対する開発」や「障害対応」などをそれぞれサービスとしてわけ、その中に「プロダクトオーナー」や「スクラムマスター」「エンジニア」などの役割がある感じになっています。

:::details どこ発？
### どこ発？
IT職における「ロールアサイン」はそれほど知られた概念ではありません^[一般的な概念ならインターネット検索をすれば組織戦略系のオウンドメディアとかが引っかかるんですが、ロールアサインは引っ掛からなかったです。なので多分そんなに広まってる概念ではなさそう]。直接的にはKADOKAWA社のDXの中で取り入れられていたらしく、弊社もKADOKAWA社のロールアサインを参考に進めているようです。

もりたの推測ですが、多分ITとば別文脈のどこかで似た取り組みは存在しており、それが経産省のDX推進関連の資料の中で言及^[https://www.meti.go.jp/policy/it_policy/jinzai/skill_standard/20240708-gp-1.pdf] され、そこからKADOKAWA社のDX事業の中でエンジニアチーム向けに調整されたのではないかと思っています。
現時点で正式な情報源として参照できそうなのは、そのKADOKAWA社でDXを進めていた各務さんという方が執筆された書籍[『世界一わかりやすいDX入門』](https://amzn.asia/d/0eVrt1Dk)がよかろうと思います。

また以下のような記事もあるので、読んでみていただけると雰囲気が伝わるかなと思います。
https://internet.watch.impress.co.jp/docs/special/kdx/1461268.html

:::
### 具体的なロールの種類
ロールの種類は以下です。おそらくこれは固定ではなく、組織によって可変なものだと思います。

| ロールの名前        | 何をする人か                    |
| ------------- | ------------------------- |
| ストラテジスト       | プロダクトを横断してROIを高める人        |
| コンサルタント       | （業務観点で）全体のサービス設計に責任を持つ人   |
| ソリューションアーキテクト | サービス横断的にアーキテクチャについて見れる人   |
| プロダクトオーナー     | 対象の施策のROIを高める人            |
| スクラムマスター      | スクラムを実現する上での障壁を取り除く人      |
| アーキテクト        | （システム観点で）全体のサービス設計に責任を持つ人 |
| エンジニア         | システムの開発と構築に責任を持つ人         |
| オペレータ         | サービスの運用に責任を持つ人            |

## どうやっているのか
### 実際の手順
見えている範囲では、以下のような流れて導入されていました。

- ロールアサインの解説
- サービスの洗い出し
- ロールの設定
- 各担当者による方針の決定・運用
- 実行のためのリソース調整など

サービスの洗い出し・ロールの設定はスプレッドシートで役割を整理しています。

### 大事っぽいこと
もりたはチームメンバーの立ち位置でこれに乗っかっていただけなんですが、やる上で大事だろうなと思ったことは以下です。

- 大義名分があること
	- ロールアサインでは、開発チームの持っているサービスが可視化され、それぞれのロールが設定されます。そのロールの中には、それまでやりたかったけど日々の忙しさから着手できなかった種類のロール（プロダクトオーナーやコンサルなど）が含まれています。
	- ロールアサインを導入することで、これらの本当はやりたかったけど現状提供する価値を毀損してまでやるべきなのか判断できていなかったことが、突然やってもいいタスクになります。
	- 新しい活動に取り組むのって一人でやろうとすると勇気がいるんですが、その時に組織が大義名分を用意してくれるととってもやりやすいな〜〜と思います。権力によるサポートがなくても下から新しいことを始めればいいじゃん、みたいなのはそうなんですが、組織としてはこういうカルチャーとしてあった方がポジティブなんじゃないかと思います。今後もこういった動きが続くと良いなと思っています。
- 数字を出すこと
	- ロールアサインのメリットは、チームの提供するサービスがなにで、その遂行に必要なロール・リソースの過不足が明確になることにあると考えています。特に前者の提供サービスの可視化という観点では、その可視化したサービスがどれほどの価値を持つのかというのを明らかにしていくことでリソース配分の妥当性などを判断する材料になりそうだと考えられます
	- そうした時に、各サービスのプロダクトオーナーが数字で（つまりエンジニアチームの活動を売り上げなどのビジネス上の指標で）表現できることが重要になってきます。ここまでやれるとチームの価値がどこにあるのかを語りやすくなるのかなと思います

## なにがいいんだ
まだ開始した直後なので振り返りには早いのですが、現時点で心躍らせている点としては以下があります。大事っぽいことと若干かぶってしまいましたがご容赦ください。
### リソース配分が明確になる
これまでは何となくこういう仕事があるよね、というのがあり、その中でどれくらい時間を割くのかは個々人に任されていました。
ただ、ロールアサインによってサービスとその価値が可視化されることで、自分たちの行う仕事のリソース配分に対して納得感を持つことができるようになるのではないかと考えています。
### ロールの解像度が上がる
「大事っぽいこと」の節でも触れていますが、これまでもプロダクトオーナーやコンサルのような動きは局所的に発生していたり、またはやりたいなと考える人はいました。ただ、チームの中で役割としてそういったロールがあったわけではなく、何となく下流から仕事を始め、慣れてきたら上流まで手をだす余裕ができる程度の粗い捉え方だったと感じています。上流に近くなるほどチームリーダーが担当することが多く、そもそもメンバーからはリーダーのやっている仕事（＝ロール）がブラックボックスでありアクセスしがたいという状態は、いろんな組織であるあるなのではないでしょうか。
それに対してロールアサインの良いところは、チーム内の役割が可視化され、担当可能なものとしてオープンになることです。
