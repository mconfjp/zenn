## はじめに
今回から「ORMで学ぶソフトウェア産業四半世紀のベストプラクティス（仮）」と題して全5回の連載を始めます。連載のとっかかりとして、まずみなさんに大きな疑問を提示したいと思います。「ORMは必要なのか？」という疑問です。

### ORMって必要ですか？
ORMが必要か？　というテーマは、SNSや記事投稿コミュニティでも繰り返し議論されてきました。
実情としては批判的な意見が目立ちます。発行されるSQLがブラックボックスになることや、初心者でない限りはそもそも必要がないといった指摘が散見されます。肯定的な意見も、その裏返しとして「ビギナーに優しい」「初学者の教育には良い」というものが中心です。
この初心者向けというイメージには、筆者も身に覚えがあります。たとえば初めてWeb開発を学んだ際は、Webフレームワークに組み込まれたORMを何気なく使っていました。SQLが書けなくてもORMがあればデータベースにアクセスできることは、初学者にとって心強いものです。こうした「役に立ったときのイメージ」が、よくも悪くも強く残っているのではないでしょうか。
つまり世間一般のイメージとして、ORMはしばしば「SQLを苦手とする人でも直感的にDBにアクセスできる、SQL生成のためのツール」だと思われがちです。
実際に、コードを見てみましょう。

リスト1 - Eloquentによるコード例
```php
$user = User::with('address')->find(1);
echo $user->name;
echo $user->address->city;
```

リスト1はPHPのEloquentによる実装です。処理は抽象化され、短い記述でデータを取得できます。
これをEloquentなしで書き直すことも可能です。現代であればAIに簡単な指示を出すだけでこのように書き換えてもらえます（リスト2）。

リスト2 - Eloquentを使わないコード例

```php
$stmt = $pdo->prepare(
    'SELECT u.*, a.city FROM users u
     LEFT JOIN addresses a ON a.user_id = u.id
     WHERE u.id = ?'
);
$stmt->execute([1]);
$row = $stmt->fetch(PDO::FETCH_ASSOC);

echo $row['name'];
echo $row['city'];
```

では、ORMの利用をやめるために、このAIによる変換コードをそのままプロダクトに入れられるでしょうか？　実際のところ、それは難しいでしょう。なぜなら、ORM脱却の影響範囲がわからない——つまり、ブラックボックスとなったORMにどれだけの技術が詰まっているのか、判断がつかないからです[^4]。

ここで2つの文章を引用したいと思います。

>  オブジェクトリレーショナルマッピングは多くの人にとって避けられない話題なので、この題材に関する文書は多くある。しかし意外なことに、一貫した、完全な、最新の書籍がない

> 我々のゴールは頑健かつ効率的な永続化データの管理にあることを覚えておこう

前者はマーティン・ファウラー著『エンタープライズアプリケーションアーキテクチャパターン』からの引用です。現代につながるORMの議論を先導したマーティン・ファウラーは、このように「ORMの議論はまとまっていない」と言及しています。その状況は、彼の発言から四半世紀が過ぎようとする現代でも、形を変えて残っています。

後者はJava製のORMであるHibernateを作り上げたGavin KingとChristian Bauerによるもので、著書『Hibernateインアクション』からの引用です。ORMを利用する目的について整理する中で、「頑健かつ効率的な永続化データの管理」がゴールであると述べています。

筆者もこれらの意見に同意します。現代のITシステムにおいて、アプリケーションとリレーショナルデータベースは鉄板中の鉄板アーキテクチャです。それらをつなぐORMは、「初心者向けのSQL生成機」ではなく、さまざまな形で我々のシステムを支えています。そして、その安易な印象を支えてしまっているのが、ORMをめぐる議論の未整備さにあると考えます。

以上のような課題感と共に本連載を始めます。
第1回の今回はORMの歴史と、ORMの取り扱う課題について学びます。ORMのたどってきた歴史を振り返ることでORMがどんな文脈の上に存在したのかを考え、ORMが解決している課題を広く掴むことを目的とします。
## ORMと歴史
ではORMが現代に至るまでにたどってきた歴史から振り返っていきましょう。ORM登場までの流れを追い、ORMがどのような課題にアプローチしてきたかを確認します。あわせて、この時代の主要な議論や資料も整理します。
ORMの歴史は3つのまとまりに分けて語ることができます。「RDBとインピーダンスミスマッチ」「初期ORMの成立」「現代につながるようなORMの潮流」です。
### RDBとインピーダンスミスマッチ
表1 - RDBとインピーダンスミスマッチ

| 年     | 出来事                                                                                  |
| ----- | ------------------------------------------------------------------------------------- |
| 1963年 | GeneralElectric社がネットワーク型DBのIDSを発明                                                     |
| 1968年 | IBM社が階層型DBのIMSを発表                                                                     |
| 1970年 | E.F.コッドによる『A Relational Model of Data for Large Shared Data Banks』発表。RDBの理論的な出発点となる論文 |
| 1979年 | Oracleが世界初の商用RDBとしてOracle2を発表                                                         |
| 1984年 | OODBの論文『Making Smalltalk a Database System』の中でインピーダンスミスマッチという語が使われる                   |

データベースの歴史は1960年代のネットワーク型・階層型DBから始まります。1963年にGE社がネットワーク型DBのIDSを発明し、1968年にはIBM社が階層型DBのIMSをリリースしました。そして1970年、IBMの研究員E.F.コッドが関係モデルの理論を論文として発表。この理論が製品として実現したのは1979年のOracle2が初めてのことでした（表1）。

そして1984年、ORMを語るうえで重要な論文が提出されます。それが『Making Smalltalk a Database System』です。この論文自体はOODB（Object-Oriented Database）製品について論じたものですが、ネットワーク型・階層型・リレーショナル型といった既存のDBを相手どり、データベース一般とプログラミング言語一般の間には「インピーダンスミスマッチ」が存在すると指摘しています。「インピーダンスミスマッチ」とはもともと電気・電子工学領域の用語で、信号の送り手と受け手のインターフェースが合致しないことによる不整合やエネルギーのロスを指します。この論文ではDBとプログラミング言語の間にある問題を比喩的に「インピーダンスミスマッチ」と表現したわけです。そしてこの言葉はその後、特にRDBとOOPの間にある不整合、つまりはORMが解決する不整合を指す用語として普及します。

同時に重要なのは、当時はインピーダンスミスマッチの解決方法としてOODBなどが有力候補とみなされており、ORMはまだ商用製品として登場していなかったということです。では実際のところOODBの登場で問題は解決したのか……というと、みなさんご存じのとおり、解決しませんでした。DBの覇権を取ったのはRDBであり、その後OOPが成熟すると、ORMが現実的な解決策として登場します。

### 初期ORMの成立

表2 - 初期ORMの成立

| 年     | 出来事                                                                                                             |
| ----- | ---------------------------------------------------------------------------------------------------------------- |
| 1994年 | TopLink for Smalltalk, ObjectLensの発表                                                                             |
| 1996年 | Java 1.0公開                                                                                                       |
| 1996年 | TopLink for Java登場                                                                                               |
| 1997年 | 『Mapping Objects to Relational Databases: O-R Mapping』『Foundations of Object Relational Mapping』といったテクニカルペーパーの発表 |

初めての商用ORMが登場したのは1994年のことでした。最初期の製品としてTopLink、ObjectLens[^1]が挙げられます。当初Smalltalk向けにリリースされたTopLinkでしたが、1996年のJava 1.0リリースを受けJava向けにも移植されます。また、その翌年には『Mapping Objects to Relational Databases: O-R Mapping』『Foundations of Object Relational Mapping』といったORMの基礎を整理するような資料が登場しています（表2）。前者はオブジェクトとリレーショナルモデルの間にある構造的なギャップについて体系的に整理したものです。後者はオブジェクトとリレーショナルモデルの特性を整理したうえで、RDBでオブジェクトを扱うための方法を理論的に提案するような文書です。
1990年代はOOPとRDBというソフトウェア開発における覇権的な要素がそろい、そのギャップにORMという現実的な解決策があてがわれた時代でした。一方で議論はまだ成熟し切っておらず、RDBにオブジェクト関連の機能を持たせようとするなど、別の方策はまだまだ検討されていました。
### 現代につながるようなORMの潮流
表3 - 現代につながるようなORMの潮流

| 年     | 出来事                                                 |
| ----- | --------------------------------------------------- |
| 2001年 | OSS製ORMとしてHibernateが登場し、爆発的に普及                      |
| 2001年 | アジャイルソフトウェア開発宣言                                     |
| 2002年 | 『Patterns of Enterprise Application Architecture』出版 |
| 2003年 | エリック・エヴァンスの『Domain-Driven Design』出版                 |
| 2004年 | Ruby on Railsの一部としてActiveRecordが登場する                |
| 2006年 | ORM標準仕様であるJPA（Java Persistence API）が策定される         |

ではいつごろからORMが決定的な存在となったのかというと、わたしは2000年代に入ってからだと考えています。特に2000年代前半はORMにとってそれまでの議論が具体的なアウトプットとして出てくる時代でした（表3）。
まず、ORMのエポックメイキングな製品としてHibernateとActiveRecordが登場しています。前者はOSS製ORMとして爆発的に普及した製品、後者はその後のORMの一大流派を作る製品です。また議論の成熟という観点では、『Patterns of Enterprise Application Architecture』の出版や、標準仕様であるJPAの策定もこの時代です。1994年に登場したORMが、数年にわたる議論の蓄積を経て、現代につながる技術として確立した時代だととらえられるでしょう。

ORMを巡るもう1つの流れとして外せないのがDDD（ドメイン駆動設計）とアジャイル開発プロセスの存在です。両者は20世紀の重厚なソフトウェア開発に対してのアンチテーゼとして登場し、現代までの四半世紀を支配するソフトウェア開発における重要なパラダイムです。『Patterns of Enterprise Application Architecture』にてORMの理論的な成熟に大きく関与したマーティン・ファウラーは2001年のアジャイルソフトウェア開発宣言に参加していますし、『エリック・エヴァンスのドメイン駆動設計』の序文を書いています。また、DDDが問題にしたのは、それまで当然とされていた「分析モデルと実装コードは別物」という開発スタイルであり[^2]、モデルと実装を一致させようとすれば、ドメインオブジェクトをRDBに永続化するための橋渡し役が必要になります。ORMはまさにその役割を担うものとして、DDDの文脈の中に位置づけられると、筆者は考えています。
### 歴史のまとめ
ORMの歴史をまとめましょう。
この章ではインピーダンスミスマッチ、初期のORM、そして現代のORMについて取り扱いました。ソフトウェアの歴史の中で、永続化レイヤーとアプリケーションはさまざまな製品・手法が試みられてきました。その中でOOPとRDBが現代では覇権をとり、その間を結ぶ存在としてORMが現実的なミドルウェアとして存在します。
また、現代のORMがどのような背景から出てきたのかも大きな観点です。90年代の議論を経て、2000年代のDDDとアジャイルの流れの中で、現代につながるORM製品は出てきています。
このように過去の課題意識を学んでみると、現代の我々が見ている光景も変わるのではないでしょうか。現代は、この四半世紀続いたDDDとアジャイルというソフトウェア開発のパラダイムが、AIの登場によって再編成される時代です。ビジネスの変化に対して、システムには一層高速な追従が求められます。そのような時代に、DDDとアジャイルの中で生まれたベストプラクティスであるORMを学ぶことは意義深いはずです。

## ORMの解決する課題
では実際のORMが解決した課題について詳しく見ていきましょう。
ここではOOPとRDBの間にある課題を広く理解することでORMがただのSQL生成機ではないことを確認し、OOPからRDBをよりよく利用する際の観点を身につけることを目的としています。
### インピーダンスミスマッチ
ORMの解決する課題は一般にインピーダンスミスマッチと呼ばれます。
歴史解説パートで説明したとおり、この単語はもともと電気電子分野の用語だったものを、1980年代のOODB開発者がデータベース一般とプログラミング言語一般の間にある乖離として引き入れた言葉でした。それが時代を経た結果として主にOOPとRDBのギャップとして使われるようになっています。
ここで注意して扱いたいのは「インピーダンスミスマッチがある＝OOPとRDBは相性が悪い」という認識についてです。これには2つの観点で反論することが可能です。

1つ目は、インピーダンスミスマッチがあることがすなわちOOPとRDBの相性の悪さを意味しないということです。後述する通り、確かにOOPとRDBの間にはデータ構造の乖離があります。しかしこれは構造的な観点に着目した場合の話です。OOPにおけるオブジェクトは特定のビジネスコンテキストの表現であり、現実のアプリケーションは複数のビジネスコンテキストを持ちながら動き続けます。RDBの二次元テーブルはアクセスパスを限定しない設計になっており、様々なクエリを書くことでそれぞれのビジネスコンテキストに対応できます。この観点では、OOPとRDBは現代のアプリケーションが必要とするものに対してよく整合しています。ORMが解消するインピーダンスミスマッチとはそのようなラストワンマイル的な乖離です。むしろ、２つのコンポーネントが協調しているからこそ、このギャップは際立って見えているのではないでしょうか[^3]。
このラストワンマイルなミスマッチについては、構造的ミスマッチと称して以降で解説します。

2つ目は、OOPとRDBの間にあるミスマッチのいくぶんかはそれらの置かれたアーキテクチャ構造からきているという点です。
ミスマッチの中には、クラサバ構成時代から続く永続化レイヤーとアプリケーションレイヤー自体の分離によって引き起こされるものも多く存在します[^5]。こちらは振る舞いのミスマッチとして説明をします。

では構造的なミスマッチと振る舞いのミスマッチについてそれぞれ見ていきましょう。
### 構造的なミスマッチ
構造的なミスマッチとは、OOPのデータ構造とRDBのデータ構造の間にあるギャップです。
OOPにおけるオブジェクトとはリスト3のような形をとっていることが多いです。

リスト3 - オブジェクトの例

```
// オブジェクトの例。疑似コード
User {
  id: 1,
  name: "山田太郎",
  address: Address {
    prefecture: "東京都",
    city: "千代田区",
    street: "千代田1-1"
  }
}
```

対して、RDBの扱うリレーションは2次元データです（リスト4）。

リスト4 - テーブルの例

```
// usersテーブル
| id | name     |
|----|----------|
| 1  | 山田太郎 |

// addressesテーブル
| id | user_id | prefecture | city   | street    |
|----|---------|------------|--------|-----------|
| 1  | 1       | 東京都     | 千代田区 | 千代田1-1 |
```

たとえばRDBにクエリした場合、RDBから取得した2次元データをオブジェクト上に展開しなければなりません。RDBにおけるテーブルをOOPにおけるクラスに、RDBにおけるカラムをOOPにおけるプロパティに置き換えて変換していくことになります（リスト5）。

リスト5 - OOPとRDBとのデータ構造の対応関係[^6]

```
// テーブル  →  クラス
// カラム    →  プロパティ
// 行        →  インスタンス
// 外部キー  →  ネストしたオブジェクト参照
```

これが**マッピング**であり、ORMとはつまりマッピングでこのミスマッチを解消しようというものです。
ここまで見ると、ミスマッチとはいえ単純な変換処理で解消できる問題に思えます。しかし実運用してみると、いくつかの例外に見舞われることになります。代表的なものとして「継承」「粒度」「関係性」が存在します。
#### 継承のミスマッチ
まず継承のミスマッチとは、OOPに存在する継承がRDBには存在しないことを指しています。
#### 粒度のミスマッチ
次に粒度のミスマッチはオブジェクトとして扱いたいデータの粒度とRDB上で扱うデータの粒度が違うことを指します。たとえば住所を扱う際に、コード上のある地点では「東京都千代田区千代田1番1号」を1つの住所と扱うのに対して、RDBのAddressテーブルでは都道府県、市区町村、番地などに分けて保存している、といったことがあります。
#### 関係性のミスマッチ
最後が関係性のミスマッチで、RDBにおいてテーブル同士にリレーションが存在する場合、基本的にどちらのテーブルからでも参照することが可能です。しかし、オブジェクトの参照には基本的に方向性が存在します。

これらの問題は基本的に現代のORM製品によって解決されているため、私たちが頭を悩ます機会は少ないでしょう。

### 振る舞いのミスマッチ
続いて振る舞いについてのミスマッチです。これはOOPとRDBのミスマッチであると同時にアプリ（クライアント）とDB（サーバー）が別プロセスで境界越しに通信する、という構造から生じる問題でもあります。
#### 同一性の問題
メモリ上に存在するオブジェクトはメモリ上の位置などを基準に「同じインスタンスか」という同一性を判断します。一方RDBは行を一意に識別する主キーで同一性を判断しており、アプリケーション側で同一行を2回読み込んだ場合に別インスタンスが生まれてしまう問題が起きえます。これを防ぐのがIdentityMapというパターンで、一度読み込んだオブジェクトをキャッシュしておくことで同一行は必ず同一インスタンスを返すしくみをORMが提供しています。
#### トランザクションの問題
アプリケーションのメモリ上で行われる処理は、必ずしもすべてRDBに反映されるべきものではありません。この間、どのオブジェクトが変更されたかを追跡し、最後にまとめてSQLを発行するしくみがUnit of Workです。変更のたびに即座にSQLを発行するのではなく、トランザクション終了時に必要最小限のSQLをまとめて実行することで整合性と効率を両立します。
#### ロードのタイミング問題
RDBとアプリケーションは別プロセスであり、データの取得にはネットワーク越しのI/Oが伴います。関連オブジェクトを必要になった時点で初めてロードする遅延ロード（LazyLoad）か、最初にまとめて取得する先行ロード（EagerLoad）かの選択は、パフォーマンスに直結する重要な問題です。ORMはこのロード戦略を制御するしくみを持っており、N+1問題のような落とし穴もこの文脈で語られます。
### 課題のまとめ
ここまで見てきたとおり、ORMが向き合うインピーダンスミスマッチには「構造的なミスマッチ」と「振る舞いのミスマッチ」という2つの側面があります。前者はオブジェクトと2次元テーブルの形の違いであり、後者はメモリとディスクの間にあるデータ保存タイミングの違いから生じます。これらをORMが吸収しているからこそ、私たちは「オブジェクトを保存する」「オブジェクトを取り出す」という自然な操作だけに集中できているのです。

なお、ORMが担っているのはマッピングだけではありません。コネクションプールによるDB接続の効率的な管理、スキーマのバージョン管理を行うマイグレーション、クエリ結果のキャッシュやRead/Writeの分離といった機能も、多くのORM製品が提供しています。これらの恩恵を受けながら、私たちは意識することなくORMの敷いたレールの上を走っています。

## おわりに
今回はORMの歴史と、解決している課題を広く見てきました。
歴史を振り返れば、現代的なORMが生まれたのはここ四半世紀のことであり、DDDやアジャイルというソフトウェア開発のパラダイムと共に育ってきたことがわかりました。課題の面では、OOPとRDBの間には構造と振る舞いの双方にまたがるミスマッチがあり、ORMはそれをまるごと引き受けている存在だということを確認しました。
ORMは初心者向けの便利ツールではなく、この四半世紀のベストプラクティスが凝縮されたミドルウェアです。次回以降は、ここで掴んだ全体像をもとに各論へと踏み込んでいきます。あらためてHibernateの著者の言葉を胸に刻んでおきましょう。

> 我々のゴールは頑健かつ効率的な永続化データの管理にあることを覚えておこう

## 今後の予定
全5回の本連載は、次回以降、ORMをめぐるよくある疑問を入り口にしながら、より深いテーマへと踏み込んでいきます。第2回は構造のマッピング、第3回はJOINとN+1問題、第4回はロックとUnit of Work、第5回はマイグレーションとActiveRecord/DataMapperパターンを取り上げる予定です。
