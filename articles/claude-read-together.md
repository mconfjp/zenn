# 本文

## これはなに

こんにちは、レバテック開発部のもりたです。
みなさんはテックな情報のインプット、どのようにやっていますか？　もりたも日頃からWeb記事なりを読もうとしていますが、Claudeと一緒に記事を読む習慣を作ったことでだいぶ捗るようになりました。この記事ではその体験とスキルを紹介します。
### 構成


## 課題

### 元々のインプット習慣

もりたのインプットはWeb記事中心です。気になっていることを検索して調べていくスタイルで、Xで流れてきた記事を読むこともあります。

ナレッジ管理にはObsidianを使っていて、Zettelkasten方式でノートを蓄積しています。気になる記事があればObsidian Web Clipperで記事をObsidian上に保存し、clip-processという自前のスキルで自動的に要約・翻訳されるようにしています。英語記事が多いので、日本語要約を先に読んでおくと読む前のハードルが下がります。

clip-processについては後述します。

### 今回の課題

この仕組みを整えても、次の課題が残っていました。

- **WebClipが溜まっていくばかりで読めない**：要約まで用意されているのに、読もうとすると後回しにしてしまう
- **読もうと思っても途中で挫折する**：難しい内容が出てくると集中力が切れて読むのをやめてしまう
- **読めてもそれで終わり**：読んだ気になっても身についた感じがない。インプットが次のアウトプットに繋がらない

## 解決策

### Claudeと一緒に読むようにした

Claudeと一緒に記事を読むようにしました。具体的なフローはこうです。

1. **印刷して読む**：PCから離れて紙で読む。デジタルデバイスから離れることで集中できる
2. **わからなかった単語をClaudeに質問**：読み終わったあと、気になった用語や内容をClaudeに聞く
3. **記事の内容についてClaudeと議論**：主要論点をClaudeと整理して、理解を深める
4. **Claudeから内容の質問**：Claudeが理解確認の質問を3問出してくれる
5. **感想をまとめる**：感想をClaudeに話すと、Obsidianに感想ノートとして保存してくれる
6. **Permanentノートにできるものはする**：感想をもとに、永続的な知見として昇格させる

### 実際のスキル

このフローはClaude Codeのスキルとして定義しています。スキルとは、よく使う一連の操作をMarkdownで記述しておいてコマンドで呼び出せる仕組みです。

#### フォルダ構成

スキルは以下のフォルダ構成を前提にしています。

```
zettel/
└── 1_literatures/
    ├── Clippings/           ← Obsidian Web Clipperの保存先
    ├── 『タイトル』本文.md
    ├── 『タイトル』翻訳.md   （英語記事のみ）
    └── 『タイトル』要約.md
```

#### clip-processスキル

WebClipした記事を処理するスキルです。`Clippings/` 内のファイルを読み込み、英語記事なら日本語翻訳と要約ノートを自動生成します。処理後は `1_literatures/` に整理されます。`/clip-process` で起動します。

````markdown
---
name: clip-process
description: ObsidianのWebClip記事を処理し、要約ノートを作成するスキル（英語記事は日本語翻訳も行う）
allowed-tools: Read, Glob, Bash, Write, AskUserQuestion
---

## タスク

1. **処理対象ファイルを確認する**
   - `1_literatures/Clippings/` 内のファイル一覧をGlobで取得する
   - すでに処理済みのファイル（`本文`・`翻訳`・`要約` で終わるもの）は除外する
   - 候補一覧をユーザーに提示し、どのファイルを処理するか確認する

2. **ファイルを読み込んでタイトルを確定する**
   - 対象ファイルを読み込む
   - frontmatterの `title` フィールドを取得する
   - `title` がない場合はファイル名から推測してユーザーに確認する

3. **記事の言語を判定する**
   - 本文の大部分がひらがな・カタカナを含む → 日本語記事
   - 本文が主に英語 → 英語記事
   - 判定が難しい場合はユーザーに確認する
   - 以降で「英語記事のみ」と書かれたステップは日本語記事の場合スキップする

4. **元記事をリネームする**
   - Bashで `mv` を使い、元ファイルを `『タイトル』本文.md` にリネームする

5. **日本語翻訳ノートを作成する（英語記事のみ）**
   - ファイル：`1_literatures/Clippings/『タイトル』翻訳.md`
   - 元記事の本文全体を日本語に翻訳する
   - 見出し構造・段落構成を保持する
   - 固有名詞・技術用語は原語を括弧内に残す（例：「オブジェクト関係マッピング（ORM）」）

6. **要約ノートを作成する**
   - ファイル：`1_literatures/Clippings/『タイトル』要約.md`
   - 英語記事の場合：

```
---
status: unread
---

# これはなに
（この記事が何であるかを1〜2行で説明）

## 関連ファイル
[[『タイトル』本文]]
[[『タイトル』翻訳]]

# 要約
（記事の内容を日本語で要約）
```

   - 日本語記事の場合：

```
---
status: unread
---

# これはなに
（この記事が何であるかを1〜2行で説明）

## 関連ファイル
[[『タイトル』本文]]

# 要約
（記事の内容を日本語で要約）
```

7. **ファイルを移動する**
   - Bashで `mv` を使い、`Clippings/` から `1_literatures/` に移動する

8. **完了を報告する**
   - 作成・移動したファイルのパスを列挙して報告する
````

#### read-togetherスキル

記事を一緒に読むスキルです。`1_literatures/` 以下の要約ノートを一覧表示し、選んだ記事について7フェーズのセッションを進めます。`/read-together` で起動します。

パスはご自身のObsidianのvault構成に合わせて変更してください。

````markdown
---
name: read-together
description: 資料を一緒に読むスキル。1_literatures内の資料ファイルを対象に「選択 → 読書 → 理解 → 確認テスト → 感想ノート → 読了」の学習セッションを実施する。「一緒に読もう」「この資料を読む」「資料を読みたい」など、資料を読む作業を始めるときはこのスキルを使う。
allowed-tools: Read, Glob, Bash, Write, Edit, AskUserQuestion
---

## ステータス定義

資料ファイルの frontmatter `status` フィールドで読書進捗を管理する：

| 値 | 意味 |
|---|---|
| `unread` | 未着手（または status 未設定） |
| `reading` | 読書中（Phase 1〜2） |
| `processing` | 全体理解中（Phase 3） |
| `reviewing` | 確認テスト中（Phase 4） |
| `reflecting` | 感想ノート作成中（Phase 5） |
| `harvesting` | permanentノート化待ち（Phase 6） |
| `done` | 完全読了（Phase 7） |

---

## Phase 1: 資料の指定

1. `1_literatures/` 以下の `.md` ファイルを Glob で取得する
   - 除外: `*本文.md`, `*翻訳.md`（これらは生素材であり読書の対象ではない）
2. 各ファイルの frontmatter から `status` を読み取る
3. 優先順位をつけてユーザーに一覧を提示する
   - 優先: `status: unread` または status 未設定
   - 次点: `status: reading` / `processing` / `reviewing` / `reflecting`（途中再開）
   - 末尾: `status: done`
4. ユーザーが資料を選択したら：
   - ファイルを Read で読み込んでおく
   - frontmatter に `status: reading` を設定する
     - frontmatter がある場合: Edit で `status:` 行を追記または更新
     - frontmatter がない場合: ファイル先頭に以下を挿入する

       ```
       ---
       status: reading
       ---
       ```

---

## Phase 2: 読書

ユーザーに伝える：「資料を読んでください。読みながら気になった点があれば聞いてください。読み終わったら教えてください。」

質問が来たら**聞かれたことだけ**に答える。1〜3文で簡潔に。
- 文脈から先読みして関連情報を補足しない
- 資料全体のまとめや意味づけは Phase 3 に持ち越す

ユーザーから「読み終わった」などの合図が来たら：「次は一緒に内容を整理しますか？」と確認してから Phase 3 へ。

---

## Phase 3: 全体の理解

1. `status: processing` に更新する
2. Claude は読み込んでいる資料をもとに**主要論点を整理して提示**する
3. ユーザーの理解を引き出す：「この点はどう理解しましたか？」「気になった部分はありましたか？」
4. ユーザーとディスカッションする
5. ユーザーが「理解できた」などの合図を出したら Phase 4 へ。

---

## Phase 4: 確認テスト

1. `status: reviewing` に更新する
2. 資料の内容に基づいた**理解確認の質問を 3 問**出す
   - 1問ずつ出す
   - 事実確認だけでなく、理解の深さを問う問いにする（例：「なぜ〜なのか」「〜と〜の違いは何か」）
3. 各回答に対してフィードバックする
4. 3問終了後、「感想をまとめましょうか？」と確認してから Phase 5 へ。

---

## Phase 5: 感想ノート作成

1. `status: reflecting` に更新する
2. ユーザーに感想を聞く：「この資料を読んでどう思いましたか？気づきや印象に残ったことを教えてください。」
3. ユーザーの感想をもとに感想ノートを作成する
   - **ユーザーの言葉をそのまま使う**
   - Claude が観点を補足したい場合は `## Claude先生より` セクションに分けて書く（任意）

**感想ノートのフォーマット**（`1_literatures/YYYY.M.D_『タイトル』感想.md`）：

```markdown
---
title: 『タイトル』感想
date: YYYY.M.D
---

# 感想

（ユーザーの言葉をそのまま記録）

## Claude先生より

（Claude が補足・コメントを加える場合のみ。なければ省略）
```

4. 元の資料ファイルの末尾に感想ノートへの逆リンクを追記する
5. 完了したら Phase 6 へ。

---

## Phase 6: permanent化待ち

1. `status: harvesting` に更新する
2. 作成・更新したファイルのパスを列挙して報告する
3. 「感想ノートをもとに、permanentノートを作成してください。」と案内する
4. 作成の合図が来たら `status: done` に更新して Phase 7 へ。

---

## Phase 7: 完全読了

「お疲れ様でした！」と伝えて作業を完了する。

---

## 注意事項

- **各フェーズの移行前に必ず確認する**：「次に進みますか？」と一言添えてからフェーズを進める
- **status 更新は各フェーズの開始時**に行う：途中離脱しても resume できるよう状態を保持する
- **感想はユーザーの言葉で**：Claude が感想を代筆するのではなく、ユーザーから聞いた言葉をまとめる
````

## やってよかったこと

### ほんまに読める

印刷してわかんないところを潰しながら、Claudeと議論しながら読むとだいぶ読めます。輪読会を毎日30分くらいやっているようなもんです。

### 中途半端な状態で放置するのが極端に減った

以前は読み始めたら理解まで終わらせて、Permanentノートにするところまで一気にやらないといけないような荷の重さがありました。今の仕組みだと、まず印刷して読んで、トイレに行った隙とかにClaude Codeのリモート機能などでちょいちょいと議論を進めたり感想を書いたりというのができるので気軽に読めます。

### Claude先生からのコメントがついて嬉しい

感想ノートを作るとき、Claudeが「Claude先生より」というセクションでコメントをつけてくれます。地味にやる気出ます。

## おわりに

自分の好きなものを知るというのは本来楽しいことだと思います。ただ、それを自己研鑽として有意義なものにしようとするとちょっとハードルが高くなります。このスキルでは有意義にするまでにある課題を細かく対処することで、簡単にインプットできるようにしました。このスキルを使ってインプット習慣を作ることで、好きなものにもっとのめり込める人が増えたら嬉しいと思っています。
