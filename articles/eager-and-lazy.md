## リード文
連載の第二回はロード戦略です。アプリケーションがRDBからデータを引っ張ってくるとき、悩みの種になるのが大量のデータロードです。今回はその大量データのロードに対してORMがどんな解決策を提供するかを見ていきます。
## 第二回：ロード戦略
ORM連載の第二回です。前回の第一回では、ORMがDDDやアジャイルといった現代的なソフトウェア開発の枠組みの中で育ってきたことを確認しました。また、ORMの課題領域としてOOPとRDBの間にある構造的なマッチングの問題と、振る舞いの問題があるという点も確認しました。
第二回の今回は、ORMを利用することの多いWebバックエンドエンジニアにとっては聞き馴染みのあるLazyロードとEagerロードを体系化する中で、ORMが扱う振る舞いのマッピングについて、入り口に立ってみようと思います。

## ロード戦略とは
今回は、ORMのデータロード戦略について取り扱います。
データロードとは、クエリを発行し、RDBからデータを取得することを指します。特にJOINを利用した場合、結果データセットが複数のテーブルを結合した形となり、ロード量が大規模になることがあります。データ戦略とは、その大規模データロードに対してORMが提供する戦略のことです。
ここでは戦略の詳細に入る前に、この領域における主要な課題の共有をするところから始めます。主要な課題は2つあり、「デカルト積問題」と「N+1問題」です。
### デカルト積問題
一つ目がデカルト積問題（別名、直積問題）です。例を交えながら説明しましょう。

リスト1 - ユーザーと投稿、コメントをJOINするクエリ

```sql
SELECT *
FROM users u
JOIN posts p    ON p.author_id = u.id
JOIN comments c ON c.post_id  = p.id;
```

こちらはよくある、なんの変哲もないクエリです。ユーザーが複数のポストを投稿し、そのポストにさらにコメントがついているという構図です。
では以下はどうでしょうか。

リスト2 - ユーザーと投稿、ファイルをJOINするクエリ
```SQL
SELECT *
FROM users u
JOIN posts p ON p.author_id = u.id
JOIN files f ON f.user_id   = u.id;
```

こちらも似たようなクエリに見えますが、よく見るとリレーションのつながり方（構造）が異なります。
リスト1のクエリは、`users`（親） - `posts`（子） - `comments`（孫）という関係性です。それに対してリスト2のクエリは`posts`（子） - `users`（親） - `files`（子）という関係性です。`users`から`posts`と`files`の二つの子テーブルが生える形となっています。
これらの違いは図にするとわかりやすいです。

図1_リレーションの違い
![](https://static.zenn.studio/user-upload/7e377531c645-20260703.png)

図の右のようなクエリは、デカルト積問題と呼ばれる結果データセットに関する問題を引き起こすことがあります。
デカルト積問題を抱えたクエリは、結果データセットの大きさを予測することが難しくなります。
まず、親-子-孫関連のクエリは結果セットが孫の数と一致するため最終的なデータサイズの予測は容易です。それに対して子-親-子のクエリは結果セットの行数が特定の親レコードに紐づく子レコードを掛け合わせた数となります。そのため単一テーブルの行数から結果データセットの量を見積もることができません。それどころか、「特定の親レコードに紐づく子レコード」というのは実データの分布という把握しづらい要素の影響を受けます。
例えば上位10%のユーザーが投稿とファイルの約80%を持つ、というデータ分布を作ってみましょう。そうすると、以下のような結果データセット量となります。

リスト3 - 各テーブルのデータ量、データの分布、結合後のデータ量を示す

```
テーブルの行数:
  users    :   1,000行
  posts    :  10,000行
  comments : 100,000行
  files    :  10,000行

データ分布:
  上位10%のユーザー（100人）がpostsとfilesの約80%を保有

結合後の行数:
  リスト1（users → posts → comments）: 100,000行
  リスト2（posts ← users → files）   : 644,600行
```

この通り、リスト2のクエリは644,600行まで膨れ上がりました。リスト1のクエリが孫であるcommentsの行数そのままの100,000行に収まっているのと対照的です。
仮にpostsの各レコードに10KB程度のテキスト（日本語で3,000文字強）が乗っていた場合、RDBから受け取る生データだけで6GBを超える計算になります^[検証の詳細な内容はこちら：https://zenn.dev/levtech/articles/cartesian-product-doctrine-orm]。AWS ECS（Fargate）で1vCPU・メモリ2GBの構成を使っていたら、OOMが発生してアプリが落ちてしまいます。

これがデカルト積問題です。
なお単純にロード量が多ければアプリケーションサーバーのメモリは圧迫されます。デカルト積の発生に関わらず、大量データロードのメモリ圧迫問題には注意が必要です。

### N+1問題
続いてがN+1問題です。
こちらは複数テーブルをロードした時に、アプリケーションがどのようにクエリを発行するかにまつわる問題です。簡単な例で説明しましょう。

リスト4 - 通常のクエリ
```sql
SELECT *
FROM users u
JOIN posts p    ON p.author_id = u.id
JOIN comments c ON c.post_id  = p.id;
```

こちらは再び、よくあるクエリ発行の例です。
これに対して、N+1問題のクエリ発行はこうなります。

リスト5 - 結合を分割して個別取得するクエリ
```SQL
-- idのリストを取得する

FROM users u
JOIN posts p    ON p.author_id = u.id;

-- idリストから、以下をくり返し

select *
from comments
where post_id = $id;
```

大量のロードデータが生まれうる状況で、リレーション先を1行ずつ取得しようとしています。これを実施すると、リスト5の2つ目のクエリがIDの個数分発行され、パフォーマンスが犠牲になります。

## 対策としてのロード戦略
これらを回避するためにはどんな方法があるのでしょうか？　ORMが提供する代表的な対策がロード戦略です。
ロード戦略にはLazyとEagerという代表的な二つの手法があります。これらを選択することで、リレーション先のコレクションやスカラーの取得タイミングを調整することができます。
例えばユーザーとポストをJOINしてデータを取得したいとなったときに、Eagerでは両方のテーブルを同時に取得します。それに対して、Lazyはまずユーザーを、そしてアプリケーション上でポストのデータにアクセスしたタイミングでポストのデータを取得する、というような動きになります。

図4 - EagerロードとLazyロードのイメージ
![](https://static.zenn.studio/user-upload/d96088181775-20260709.png)

EagerとLazyは対照的な関係にあり、それぞれN+1問題とデカルト積問題を解決するものです。これは単純な整理であり、実際はORMによって細かな実装パターンが存在するため、ご自身が利用するORMのドキュメントを見ることでどんな選択肢があるのかを把握するのが良いでしょう。
ただし、いくつか共通した考え方や代表的な手法もあります。以下にピックアップして解説していきます。
### デフォルトのロード戦略はLazy
ORMの中には、リレーションの種類ごとにデフォルトのロード戦略を変えるものがあります。よくあるのは、リレーション先が1レコードに定まるToOne関連（投稿から見た投稿者など）ではEagerロードを採用するというものです。一方、リレーション先が複数レコードになりうるToMany関連（投稿から見たコメントなど）ではLazyロードを採用します。大量データロードのリスクがあるのはToMany側ですから、リスクのある箇所のデフォルトはLazyになっている、と言えます。
これはLazyロードが優れているからというよりは、安全側に倒した結果です。Eagerロードで想定外の大量データを掴んでしまうとOOMでプロセスごと落ちますが、LazyロードによるN+1問題はパフォーマンス劣化で済みます。障害の深刻度を比較して、深刻度の低い方をデフォルトにしているわけです。
### ロード戦略の代表的な設定タイミング
ロード戦略を設定するタイミングは、代表的なものが2つあります。
ひとつはモデル定義時です。ORMの多くは、テーブルをアプリケーション側のモデルとして扱うためのマッピング定義を持っており、そこでリレーションごとのロード戦略を指定できます。ここでの指定が、そのリレーションのデフォルトの挙動になります。
もうひとつはクエリ発行時です。デフォルトとは異なる挙動をさせたい場合に、クエリ単位で戦略を上書きします。
使い分けとしては、モデル定義では安全側のデフォルトを決めておき、リレーション先を確実に使うとわかっている処理では、クエリ発行時にEagerロードを個別指定する、という形が基本になります。
### Eagerロードの派生戦略
ひとくちにEagerロードといっても、実現方法は一つではありません。Ruby on RailsのActiveRecordを例に見てみましょう。
ActiveRecordには関連テーブルを扱う代表的なメソッドとして`preload`・`eager_load`・`includes`・`joins`の4つがありますが、このうちEagerロードの派生戦略と呼べるのは3つです。`preload`はテーブルごとにクエリを分け、`WHERE id IN (...)`で関連先をまとめて取得する方式です。JOINしないためデカルト積問題を回避できますが、クエリが分かれている都合上、関連先のカラムを使った絞り込みはできません。`eager_load`は一発のJOINですべてを取得する方式で、関連先のカラムで絞り込みたい場合はこちらを使います。`includes`は両者の自動切替で、基本はpreloadとして動き、関連先を絞り込みに使っていればeager_loadに切り替わります。
では残る`joins`は何かというと、これはJOINをするだけで、関連先のデータをロードしません。絞り込みのために結合するだけのメソッドであり、Eagerロードのつもりで使って関連先のデータにアクセスすると、そこで別途クエリが発行されます。「JOINすること」と「ロードすること」は別ものだ、というのはロード戦略を考える上で押さえておきたいポイントです。

## 課題の再定義と、より広い対応策
### 課題の再定義
ロード戦略の解説は以上です。ただ実際のところまだ疑問が残るのではないでしょうか。
今回はデカルト積問題のメモリ負荷対策としてLazyロードを紹介し、N+1問題のパフォーマンス問題対策としてEagerロードを紹介しています。ただ、メモリ負荷対策としてLazyロードを導入すればN+1が引き起こされ、N+1問題の対策としてEagerロードを導入すればパフォーマンス問題となります。これではイタチごっこです。

その指摘はごもっともです。EagerとLazyは要するにリスク受容の戦略であり、大量データロードのリスクをメモリで受け止めるか、パフォーマンス面で受け止めるかの違いにすぎません。大量データロード自体は解決できていないのです。

となると、課題を定義し直し、より広く対応策を考えることができそうです。当初はロード戦略を「大規模データロードに対してORMが提供する戦略」としていましたが、これは手段に着目した定義で、なんのための技術なのか表現できていませんでした。いま改めて定義し直すならば、ロード戦略とは「ビジネストランザクションのメモリ最適化戦略」と言い換えることができます。
そしてこの観点で考えたとき、次に挙げる観点でもう少し議論することができます。

- 観点
	- リスク受容ではないロード戦略の使い分け
	- メモリを解放する
	- 機能改修によりメモリの使用量を減らす
	- メモリの増強

上から3つがメモリ使用量を減らすような対策、最後がメモリ自体の増強を扱う対策です。
それぞれ見ていきましょう。
### 本質的な対応 - 使い分け
ひとつ目はリスク受容ではないロード戦略の使い分けです。
EagerロードとLazyロードには、大量ロードデータのリスクをどこに転嫁するか以外の使い分け方があります。
それは、ロード対象が使われないケースを見抜くことです。例えば冒頭に紹介したデカルト積問題が代表的ですが、子-親-子のような二股のJOINをするケースでは、一つのビジネストランザクションの中で処理が分岐し、利用するデータが異なることがあります。その場合、最初に全てのデータをロードする必要はなく、分岐したのちに利用するデータのみを追加でロードしてあげれば十分です。

リスト - 分岐後にロードしたいデータが異なるコード
```PHP
// Lazyロード

// 条件分岐によって読み込むJOIN先が変わる


```

このように条件分岐が存在するロードではLazyがリスク受容以上の意味を持ちます。
### 本質的な対応 - メモリの解放
ふたつ目はメモリの解放です。
ORMは使い終わってもリクエスト単位で対象のモデルを保持し続けることが多いです^[例外もあります。PythonのSQLAlchemyはセッション内のオブジェクトを弱参照で保持するため、アプリケーション側で参照されなくなったオブジェクトは自動的にGCの対象になります（ただし未フラッシュの変更があるものは残ります）。また、PHPのEloquentにはそもそもIdentityMapが存在しません]。これを意図的に解放することでメモリを解放することもできます。使っているORMのメモリの管理方法を把握しておくことで対策が取れます。

### 本質的な対応 - 機能改修によりメモリ使用量を減らす
三つ目は機能改修等を通じて使用するメモリ量を減らします。
代表的な手法がサーバーサイドページネーションです。RDBからのデータロード量をそもそも減らすことで、メモリ使用量を削減します。また、機能自体を改修し、ロード対象のデータを絞り込むことも有効です。
### 本質的な対応 - メモリの増強
最後の対応策がメモリの増強です。
もっとも直接的なのはアプリケーションサーバー自体のメモリ増強です。また、増強とは少し毛色が違いますが、ロードしたデータの置き場所をアプリケーションサーバーの外へ移すという方向性もあります。例えばJavaのHibernateには、ロード済みデータをセッション単位で保持するL1キャッシュに加えて、Redis（Redisson等のライブラリ経由）やEhcacheなどの外部キャッシュに保存するL2キャッシュが用意されており、データをどこに持つかを選択できます。

## 振る舞いのマッピング
ここまでロード戦略をご紹介してきましたが、『PoEAA』などの定義ではLazyロードは振る舞いのマッピングに含まれます^[PoEAAではLazyロードの代表的な実装パターン「遅延イニシャライズ」「仮想プロキシ」「バリューホルダ」「ゴースト」が掲載されてて面白いので読んでみてください]。
振る舞いのマッピングにはそのほかに「ユニットオブワーク」と「一意マッピング」が存在します。これらは、ビジネストランザクションとシステムトランザクションの間にあるギャップの解消を行うものだと私は考えています。そしてこの問題はトランザクションやロックに関する問題とも深く繋がっています。これらのテーマは次回のロックとトランザクションにて扱う予定です。
